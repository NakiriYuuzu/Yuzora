ACP 協定確認與 yuzora 實作評估

yuzora AgentZone · 2026-07-10 · 依據官方 agentclientprotocol/agent-client-protocol（schema v1 stable）與本地 @agentclientprotocol/sdk@1.2.0 實機驗證

TL;DR

體質良好，債務集中在「協定演進跟進」，不是架構錯誤。yuzora 已正確使用官方 SDK 的 ClientSideConnection（非手寫 JSON-RPC），client 義務面（session/update、permission、fs×2、terminal×5）9/9 全實作，cancel 語義符合 spec 的 MUST，lifecycle 工程與測試紀律是亮點。

但對照 stable v1 全貌有 3 個實際會出錯的問題（A 級）：stable 變體 usage_update／config_option_update 會被渲染成紅色錯誤塊、current_mode_update 的 store 接線是斷路（dead path）、45 行 session_info_update 攔截 hack 在 SDK 1.2.0 之後已是死重。另有功能缺口（session/set_mode、authenticate RPC、prompt capabilities）與版本債（SDK 1.2.1、adapter 版本、deprecated API）。建議先做一輪低風險同步（約 1–2 天），中期再補 mode／auth／附件能力。

評估結論

本次以官方 monorepo（schema v1 stable + docs）、TypeScript SDK 1.2.1 原始碼、Rust SDK 1.2.0，與 yuzora 本地 node_modules 內的 SDK 1.2.0 交叉驗證。結論分三層：

架構層：方案 A′ 是正確選擇，維持。前端 host ClientSideConnection 與官方 TS SDK 的設計意圖一致（Client interface 的回呼——permission、fs、terminal、update——本質上都是 UI 事務）；Zed 本身也是 client 活在 UI process。Rust 端只做 process spawn 與 NDJSON stdio 橋接，薄而穩。官方 Rust SDK 1.2.0 雖然存在，但已是大改寫的 builder API，遷移到 Rust host 沒有收益、只有風險。
協定層：義務面全齊，可選面落後半個版本世代。核心 prompt turn 迴圈（initialize → session/new → prompt → update 串流 → permission → stopReason）完整且正確。缺口全部集中在 1.x 期間 stabilize 的新方法（set_mode、set_config_option、list/resume/close、logout、authenticate）與新 SessionUpdate 變體（usage_update、config_option_update）。
工程層：紀律是亮點。acpConnection.deps.test.ts 用 ?raw import 斷言防止退回 deprecated scope；init timeout 先撈 stderr tail 再 kill；EPIPE 特判；孤兒 process 啟動回收；permission allow_always 記憶；cancel 時對 pending permission 回 cancelled outcome（spec 的 MUST）。~1000 行的 connection 測試涵蓋端到端 SDK bridge。

方案 A′ 資料流（現況實測）

front-end-hosted ACP client：WebView 持有協定狀態，Rust 只做 stdio 橋

一句話 takeaway：協定狀態全部活在 WebView；Rust 是無協定知識的字節橋。這與官方 TS SDK 的 Client interface 設計吻合，也是官方 adapter 生態（跑在 node/bun 子行程）最順的組合。

做對的事（keep）

  • JSON-RPC runtime 交給官方 SDK：correlation、cancel（$/cancel_request，1.2.0 已內建 PROTOCOL_METHODS.cancel_request）、zod 驗證都由 SDK 處理；acpConnection.deps.test.ts 防止退回 deprecated 的 @zed-industries/*。
  • client 義務面 9/9：fs/read_text_file 優先讀開啟中 buffer（未存檔內容對 agent 可見）、fs/write_text_file 走 CodeMirror + save IPC；terminal 五方法齊且後端有 byte-limit 處理。
  • cancel 語義正確：session/cancel 前先把 pending permission 以 cancelled outcome 收掉（spec MUST）；容忍 cancel 後遲到的 tool_call_update。
  • lifecycle 工程成熟：init 15s timeout（先撈 stderr 再 kill）、EPIPE 特判、exit 帶 stderr tail、respawn、啟動時孤兒 agent 回收、write 失敗立即斷線 reject。
  • UX 超出最小實作：slash commands popup（吃 available_commands_update）、plan 進度、diff 檢視／套用、permission allow_always 記憶、auth banner + 終端登入 fallback、pi-acp startupInfo banner。

其他關鍵語義

  • $/cancel_request（1.2.0 stabilized）：JSON-RPC 層通用取消。agent 收到 session/cancel 後會把取消級聯到它對 client 發出的 in-flight 請求（permission／terminal），接收方 MUST 回原請求（結果或 -32800 Request Cancelled）。SDK runtime 已內建處理，yuzora 不需手做。
  • _meta 擴充：協定所有型別都帶 _meta?；實作 MUST NOT 對值做假設；vendor key 建議帶命名空間（如 zed.dev/debugMode）；root 層禁止加自訂欄位；_ 開頭 method 保留給 vendor 擴充。yuzora 讀 _meta.piAcp.* 的做法符合此規範。
  • StopReason：end_turn | max_tokens | max_turn_requests | refusal | cancelled（與 yuzora 手抄一致）。
  • PromptCapabilities baseline：agent 未宣告時，client 只能送 text + resource_link；image／audio／embeddedContext（內嵌 resource）須看 agent 宣告。

A 級：實際會出錯／死重（3 項）

A1stable 變體 usage_update／config_option_update 被渲染成紅色錯誤塊

reducer 的 default case 把未知變體變成 kind:"error" 的「Unknown session update: X」transcript block。usage_update 與 config_option_update 是 stable v1 變體，且本地 SDK 1.2.0 的 zod schema 已認得它們——通知會通過 SDK 驗證、抵達 reducer、變成錯誤塊直接呈現給使用者。

證據：src/agent/acpConnection.ts:274-276（default → unknownSessionUpdate）；node_modules/@agentclientprotocol/sdk/dist/schema/zod.gen.js:2051-2053（zUsageUpdate literal 存在）。

影響：claude-agent-acp／codex-acp 新版與 pi-acp 一旦送 usage（token 用量的官方 stable 通道），對話裡就出現錯誤塊。這同時是被記錄過的「pi-acp token 用量顯示」需求的正解入口——官方已有通道，不必等 vendor 擴充。

建議：reducer 補兩個 case：usage_update 餵給 UI 顯示 token 用量（或先靜默＋存 store）；config_option_update 靜默更新。default 改為「靜默＋console.warn」——官方 extensibility 精神是未知內容不應以錯誤驚擾使用者（forward-compat）。

A2current_mode_update 的 store 接線是斷路（dead path）

reducer 對 current_mode_update 刻意回傳原 transcript（正確——它不是 transcript 事件），mode 的提取邏輯放在 store 的 appendUpdate；但 appendUpdate 在產品碼中零呼叫者（只有測試在呼叫）。實際接線 AgentBridge 只接了 onTranscript／onAvailableCommands／onSessionInfo 三條 callback。

證據：src/agent/acpConnection.ts:261-263（reducer 跳過）；src/state/agentStore.ts:321,703-704（mode 提取，僅測試觸達）;src/workbench/AgentBridge.tsx:47-54（三條 callback，無 mode）。grep 全庫確認 appendUpdate 無產品呼叫點。

影響：agent 自主切 mode（例如 claude 的 plan mode 進出）時 UI 永遠不知道；store 的 mode 欄位只在 session 建立時有值。

建議：runtime 加 onModeChange callback 接到 store（比照 onSessionInfo 模式），同時決定 appendUpdate 的去留（留作測試 helper 或刪除）。

A3interceptUnsupportedSessionUpdates 已是死重（45 行 stream hack）

這段 stream 攔截為「SDK 不認得 session_info_update」而生（0.4.5 時代成立）。但本地 SDK 1.2.0 的 zSessionInfoUpdate 已存在且全欄位 optional（title／updatedAt／_meta 皆 nullish + defaultOnError）——pi-acp 只帶 _meta.piAcp 的 payload 可直接通過 zod 驗證，由 SDK 正常派發到同一個 client.sessionUpdate。攔截與否行為等價，hack 純屬負債。

證據：src/agent/acpConnection.ts:754-797（intercept + isSessionInfoNotification）；node_modules/@agentclientprotocol/sdk/dist/schema/zod.gen.js:1985-1989（全 optional schema）。

建議：直接移除 intercept、把原 stream 餵給 ClientSideConnection；跑既有 session_info 端到端測試迴歸（acpConnection.test.ts 已涵蓋）。

B 級：功能缺口（4 項）

B1session modes 整層缺席（set_mode + mode UI）

session/new／session/load 回應中的 modes（currentModeId + availableModes）被丟棄；無 session/set_mode 呼叫路徑；面板無 mode 顯示或切換器。三個 preset agent 都有 mode 概念（claude-agent-acp 的 permission modes、pi 的模式），這是對使用者可見度最高的功能缺口。

證據：grep 全庫無 set_mode／setSessionMode；AgentZonePanel.tsx SessionHeader 僅顯示 model。

建議：newSession 存 modes → SessionHeader 加 mode selector → 呼叫 setSessionMode；與 A2 的 onModeChange 一起做成完整迴路。

B2authenticate RPC 無呼叫路徑；現行 workaround 對應的是 unstable 擴充

yuzora 收到 -32000 Authentication required 後走終端登入（開 terminal session 跑 agent 的 login 指令）再 retry——這對「CLI login／env var」型 agent 有效（pi／claude／codex 現況都能動）。但 stable v1 的 AuthMethod 只有 agent 變體（agent 自行處理認證，client 呼 authenticate {methodId}）；env_var／terminal 型其實還是 unstable。也就是說：協定的正規路徑（authenticate RPC）yuzora 反而沒有，遇到宣告 agent-variant authMethods 的 agent 會卡在 banner。

證據：grep 無 connection.authenticate 呼叫；acpConnection.ts:843-852（isAuthRequiredRpcError）→ agentStore.ts:179-195（terminal login + retry）。

建議：authMethods 含 agent-variant 時提供「透過協定認證」按鈕（呼 authenticate 後 retry newSession）；terminal workaround 保留為 fallback。

B3promptCapabilities 未讀；送出端只有 baseline（text + resource_link）

initialize 回應的 agentCapabilities.promptCapabilities（image／audio／embeddedContext）被丟棄，composer 送出的 PromptBlock 型別也只定義 text 與 resource_link。附圖、把開啟中 buffer 以 embedded resource 內嵌（對不能讀 client fs 的遠端 agent 重要）都不可能。

證據：acpConnection.ts:140-142（PromptBlock 型別）；initialize 回應只取 authMethods（620-635）。

建議：存 per-connection capabilities → composer 依 capability 開放附件類型（第一步：image）。

B4session 管理方法未跟進（list／resume／delete／close／logout）

皆為 1.x stabilized、capability-gated。現況 listSessions 靠 router 本地聚合（只看得到本次 app 生命週期內的 session）；session/resume（不 replay 的恢復）對長 session 的恢復 UX 比 session/load replay 更省。

建議：依 agent 宣告的 sessionCapabilities 漸進採用；優先順序 resume > list > delete／close。

C 級：演進債／衛生（3 項）

C1ClientSideConnection 已被官方標 @deprecated

本地 1.2.0 的 acp.d.ts 即已標註：「Prefer client(), which registers typed handlers with a single context object and supports connectWith and session helpers.」1.x 內仍完整支援，不急；但下個 major 有移除風險。

證據：node_modules/@agentclientprotocol/sdk/dist/acp.d.ts:975 前的 doc comment（實機確認）。

建議：列入中期遷移（builder client() + onRequest／onNotification）；遷移時可一併消掉 A3 的 stream 包裝層。

C2版本落後：SDK 1.2.0→1.2.1；adapter pins 0.57.0→0.58.1、1.1.0→1.1.2

pin 策略本身正確（可重現），缺的是升級節奏。SDK patch 升級低風險；adapter bump 建議搭配一次實機 smoke（gui-acceptance）。

C3手抄協定型別子集＋Record<string,unknown> 解析

acpConnection.ts:19-161 手抄了 ToolCallUpdate／PlanUpdate／PermissionOption 等型別，SessionUpdate 用 Record<string,unknown> + runtime guard 解析；SDK 有完整生成型別（types.gen）可直接 import。exitStatus 同時容忍 camelCase／snake_case（899-909）——官方 schema 是 camelCase，snake_case 容忍屬早期上游混亂的遺留。

建議：與 C1 遷移一起改 import SDK 型別，刪手抄與雙格式容忍。
