P10切換 workspace／folder 清空上一個的 ACP 狀態與紀錄P0 資料遺失Mconfirmed

現象

切到另一個 workspace 後，再切回來，前一個 workspace 的所有 session 與對話紀錄直接消失。

根因

三個設計缺陷疊加：

  • 連線是綁定首個 cwd 的 singleton：AgentBridge 用 useMemo(createStoreBackedConnection, []) 只建一次；ensureConnection 開頭 if (agent) return，永不為新 cwd 重新 spawn，所以 Rust registry 裡那條現役 agent 的 child.cwd 永遠是「最初的 workspace」。
  • recovery 每次切換都跑：AgentBridge 的 effect 依賴 [workspacePath]，每次切 workspace 都執行 recoverStaleAgents。它 agentList(cwd)（Rust list() 以 child.cwd == cwd 過濾）會撈回現役、不是殘留的 agent，接著 agentKill 它並呼叫 useAgentStore.reset() — 這是唯一的 production reset 呼叫點，會清掉所有 workspace 的 sessions。
  • sessions 未依 workspace 分流：agentStore.sessions 是單一全域 Map，AgentNavContent 不分 cwd 全部列出。即使沒有 reset，B 的 session 也會在 A 底下混著顯示。

證據

  • src/workbench/AgentBridge.tsx:12（singleton）· :31-38（effect 依賴 workspacePath）· :77（agentKill）· :84（reset — 唯一 production 呼叫點）
  • src/agent/acpConnection.ts:485（if (agent) return）· :587-590（首個 cwd spawn）· :614（newSession 有帶 per-session cwd）
  • src-tauri/src/agent_process.rs:235-246（list() 過濾 child.cwd，回傳現役）
  • src/state/agentStore.ts:72（全域 sessions）· :401-408（reset 全清）· src/state/workspaceStore.ts:72-78（setWorkspace 不碰 agentStore）

修法（分階段）

Stage A — 止血（急）：用 useRef 讓 recovery 整個 app 生命週期只跑一次（reap HMR／webview reload 遺留的孤兒），並從 recoverStaleAgents 移除 reset() 與尾端 setConnection；首跑時新連線尚未持有任何 process，撈到的才是真孤兒。可再讓連線公開 agentProcessId，把現役那條排除在 kill 名單外做雙保險。

Stage B — nav 依 workspace 分流：AgentNavContent 的 sessionEntries 過濾 session.cwd === cwd（保留所有 session 在 store），並加一個 activeSessionForCwd selector，避免 sendPrompt/cancel 打到隱藏的他 workspace session。

Stage C — 可延後：connection-per-cwd／切 cwd 重連。因為 newSession({ cwd }) 在協定層已正確 root 每個 session，只有在確認 pi-acp adapter 依賴 OS process cwd 時才需要。

測試影響：agentBridge.test.tsx:111-137 目前寫死了「清空」為預期（reset 後 sessions.size===0），必須改寫成「孤兒被 kill 但 store 保留」，並新增 A→B→A 不觸發 recovery、不 kill 現役 agent 的案例。agentNavContent.test.tsx 新增「只列當前 cwd」案例。
P3我提問的內容會被隱藏P0 資料遺失Sconfirmed

現象

送出 prompt 後，自己剛打的那句話從對話中消失。

根因

transcript 有兩個真相源。beginTurn 樂觀地把 {who:'you', text, streaming:true} 寫進 store 的 transcript；但 createAcpClientRuntime 另外自持一個 transcripts Map。每收到一則 session/update，runtime 都是從它自己那個（一開始是空的）Map reduce 出 next，再經 onTranscript → replaceTranscript 整批覆蓋 store 的 transcript。因為 pi-acp 不會回送 user_message_chunk，第一則 agent_message_chunk 進來時就把樂觀寫入的 you 區塊蓋掉了。

證據

  • src/state/agentStore.ts:549-563（beginTurn 只寫 store）· :314-325（replaceTranscript 整批覆蓋、不合併）
  • src/agent/acpConnection.ts:273（自持 transcripts Map）· :302-307（從自身 Map reduce → onTranscript）· :246-247（you 只由 user_message_chunk 進入，pi-acp 不送）
  • src/agent/fakeAcpAgent.ts:71-74（fake 直接跳 agent_message_chunk，重現丟失）· agentBridge.test.tsx:94-104（只驗 agent 訊息，故 CI 看不見此 bug）

修法

在送 prompt 時，把同一個 user 區塊也 seed 進 runtime 的 transcripts Map，這樣整批覆蓋時就會保留它。具體：於 acpConnection.ts 的 AcpClientRuntime 加 recordUserPrompt(sessionId, blocks)（push you 區塊、不觸發 onTranscript 以免閃爍），並在 createAcpConnection.prompt（:629）呼叫 connection.prompt 前先叫它。抽一個和 store textFromBlocks 同款的 helper，確保 seed 文字與 beginTurn 逐字相同。beginTurn / replaceTranscript / AgentBridge 都不動。

測試影響：agentBridge.test.tsx 加一條 transcript 含 {who:'you'} 的回歸斷言（修前失敗、修後通過）；store-only 測試不受影響。
P1點擊 new-session 有點慢，不會立刻打開視窗中Mconfirmed

根因

newSession 同步設了 connectionState:'connecting'，但 session 只在 await 完成後才進 map、才設 activeSessionId。那個 await 首次會冷啟 bunx pi-acp@0.0.31（下載＋node 啟動）＋ initialize（15s timeout）＋ newSession RPC，數秒延遲。而 AgentZonePanel 只在 activeSessionId && session 才渲染 session，否則停在 disabled 的空狀態 — connecting 狀態根本沒被渲染成「連線中」畫面。AgentBridge 也不預熱。

證據

  • src/state/agentStore.ts:138（同步 connecting）· :140-151（await 後才設 activeSessionId）
  • src/agent/acpConnection.ts:488（withTimeout startConnection 15s）· :587（spawn bunx）
  • src/app/panels/AgentZonePanel.tsx:145-149（無 pending 畫面）· :130-131（connectionState 只用於 error banner）

修法

加 樂觀「連線中」畫面：store 新增 pendingNewSession flag（放進 agentInitialState 讓 reset 一併清），newSession 同步 set true、成功／失敗兩條路徑都 set false；AgentZonePanel 在 pending 時渲染一個帶 spinner 的 ConnectingState（沿用空狀態外殼）優先於空狀態；AgentNavContent 的新增鈕在 pending 時 disabled 防重複 spawn。i18n 加 connecting / connectingHint。

不建議一起做的：在 AgentBridge mount 時預熱連線 — 會在使用者根本沒開 AgentZone 時就 spawn pi-acp，且與 recoverStaleAgents 每次切 workspace 的 kill 邏輯衝突（須先修 P10）。
P2tool-use 改為點擊展開；預設只顯示工具／做了什麼／結果前 50 字中Mconfirmed

根因

所有 block（tool/diff/perm/error/plan）都走同一個 TranscriptBlock，把 entry.text 用 whiteSpace:pre-wrap 全印出來（旁邊的 ellipsis 因 pre-wrap 只會換行、不會截斷）。appendToolCall/mergeToolCallUpdate 把 title＋每個 content chunk 全累進 text，而 title/kind/status/locations/rawInput/rawOutput 早就序列化在 meta 裡 — 展開檢視所需資料已經有了。

證據

  • src/app/panels/AgentZonePanel.tsx:786（全 block 走 TranscriptBlock）· :837-851（單一 span、pre-wrap、無截斷）
  • src/agent/acpConnection.ts:941-987（text 累加）· :1098-1112（meta 帶 rawInput/rawOutput/locations）

新增專用可折疊 ToolBlock，只有 kind==='tool' 走它，其餘 block 維持原樣。useState(expanded=false)；解析 meta 取 title/kind/status；把 entry.text 於首個換行切成 firstLine / body，收合時顯示 [kind] title · body.slice(0,50)…（單行 ellipsis），展開時顯示完整 text ＋ rawInput/rawOutput（JSON.stringify(x,null,2) 放進 mono <pre>）＋ locations。外層整塊做成一顆 <button> toggle（子元素皆非互動 span，無巢狀 button 問題），加旋轉的 ChevronDown。i18n 加 toolToggle/toolInput/toolOutput/toolLocations。無須動 store／acpConnection。

測試影響：agentZonePanel.test.tsx:273 的 toHaveLength(2) 需改（tool 折疊後其 inert 的 View action 併入 toggle）；建議新增「展開前隱藏、點擊後顯示」案例。

P5不會自動置底高頻Sconfirmed

根因

TranscriptList 的捲動容器是純 <div overflow:auto>，完全沒有 ref／scroll 處理（全 panel grep 不到 scrollTop/scrollHeight/scrollIntoView/useRef）。串流每個 token 都產生新 transcript 陣列參考，list 會重繪長高，但視窗不跟。

證據

  • src/app/panels/AgentZonePanel.tsx:691-704（容器無 ref/onScroll）· :9-16（import 無 useRef）
  • src/agent/acpConnection.ts:653-658（串流每 token 回新陣列 → effect 會觸發）

修法

TranscriptList 加 scrollRef ＋ nearBottomRef；onScroll 判斷「距底 < 80px 才算跟隨」（使用者往上捲即暫停自動置底、捲回底部恢復）；useEffect([session.transcript]) 在 nearBottom 時把 scrollTop = scrollHeight。以 session.transcript 為唯一 dep 即可（串流每 token 換參考，故文字增長也會跟）。jsdom-safe（讀值為 0、賦值 no-op）。

驗收：置底跟隨與「往上捲暫停」在 jsdom 測不到，需用 gui-acceptance 實機驗。

新功能評估

P9每建新 session 時，把首則系統資訊包成 ! info 放到 session header功能M可行 ✓
實測更正（反編譯 pi-acp@0.0.31 bundle 確認）：先前初判「pi-acp 不會主動送開場訊息、newSession 只回 sessionId」是錯的。pi-acp 確實會在 newSession 時建立開場系統資訊，並經兩條路輸出。你觀察到的「開頭列系統資訊」就是它。

pi-acp 實際行為

newSession 會呼叫 buildStartupInfo({cwd, fileCommands, updateNotice}) 產生一段 Markdown（preludeText），輸出兩處：

  • ① newSession response 的 _meta.piAcp.startupInfo ＝ 完整 preludeText（或 null）。
  • ② 主動 emit 一則 agent_message_chunk：sendStartupInfoIfPending() 於 newSession 回傳後用 setTimeout(0) 送出（註解明說是為「Zed 等 client 在 UI ready 前才 render agent 訊息」而設）。因此這份系統資訊現在就已經以第一則 agent 泡泡出現在 transcript。

內容（buildStartupInfo）：pi vX.Y.Z 版本 · ## Context（AGENTS.md）· ## Skills（全域／.agents／專案 .pi/skills 的 SKILL.md/.md）· ## Prompts（~/.pi/agent/prompts 的 /name）· ## Extensions · 選配 update notice。另有 quietStartup／quietStart 設定可關成只剩 update notice。

證據

  • pi-acp dist/index.js:1883-1902（newSession 建 preludeText、回 _meta.piAcp.startupInfo）· :1904（setTimeout(()=>session.sendStartupInfoIfPending(),0)）· :773-780（emit agent_message_chunk）· :2677-2786（buildStartupInfo 內容）· :1608-1615（getQuietStartup）
  • src/agent/acpConnection.ts:611-620（yuzora newSession 只讀 result.sessionId、丟掉 _meta）· :246-249（agent_message_chunk → appendMessage 進 transcript）· AgentZonePanel.tsx:536-679（SessionHeader 無 info 槽）· :154-235（可沿用的 soft banner 樣式）

修法（更正後更乾淨）

直接取 authoritative 來源，不靠猜首則回覆：(1) acpConnection.newSession 讀 result._meta?.piAcp?.startupInfo，經 upsertSessionMeta／新 callback 帶進 SessionState.infoBanner（emptySession 預設 null）。(2) ActiveAgentSession 在 SessionHeader 後插 <InfoBanner> soft strip，帶「!」badge、行數 clamp（首訊可能很長）。(3) 抑制 transcript body 的重複副本：因為現在有 authoritative 的 startupInfo 可比對，把「文字等於 startupInfo 的首則 agent 訊息」從 transcript 濾掉，避免 header 與 body 同時出現。(4) 尊重 quietStartup：startupInfo 為 null 就不顯示 banner。

測試影響：acpConnection.newSession 回傳型別需從 string 擴充（帶 startupInfo）或改用 meta callback，牽動 AgentConnection 介面與其 mock；agentZonePanel.test.tsx 的 session() factory 需補 infoBanner:null；fakeAcpAgent.ts 可加一則 startupInfo 便於測 banner 與 body 去重。
F1能否取得請求使用的 token 量？功能S卡上游 pi-acp

結論：逐請求 token 用量今天拿不到

verifier 實際反編譯了 pi-acp@0.0.31 的 bundle 求證：

  • ACP 協定本身沒有 usage 欄位：PromptResponse 只有 stopReason ＋ 通用 _meta?；SessionNotification.update union 也沒有 usage 變體。唯一結構化通道是 _meta passthrough。
  • pi-acp 內部其實有數字：getSessionStats() → {cost, tokens:{input,output,cacheRead,cacheWrite,total}}，但只透過 /session slash command 以人類可讀文字輸出。
  • 它每則 session_info_update._meta.piAcp 只帶 {queueDepth, running}；turn_end／agent_end 事件都沒被拿來塞 usage。
