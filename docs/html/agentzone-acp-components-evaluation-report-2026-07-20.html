TL;DR — 建議先看這裡

10 個目標元件中，8 個建議 skip、只有 1 個（message-scroller）值得從套件 adapt；plan／task／reasoning 收斂為「零新相依的自建 native block」。核心原因：這些元件幾乎都是 presentational shell（純展示殼、不吃資料模型），而 AgentZone 真正的難點在 ACP wire → transcript 的 reducer flattening（plan 丟 priority、tool status 不透明、meta 是 JSON 字串）與串流 row-key remount——這兩件事沒有任何一個元件能解決。

反直覺發現：真正高槓桿的元件反而在你的清單之外——AI Elements 的 Response（真 markdown 渲染，對應現況只支援 code/bold 的 MinimalMarkdown）、Prompt Input + Suggestion（composer 是 AgentZone 最大未評表面）、Context（usage token 分項）。建議路線：先做 Phase 0 地基（stable-id＋拆 2428 行大檔），再自建 native blocks，最後只引入 message-scroller 一個外來元件。

三個貫穿全案的關鍵事實

事實 1 這 10 個元件幾乎全是「presentational shell」，沒有一個吃 AI SDK runtime

10 路文件研究逐一驗證（含抓取 registry 原始碼）：AI Elements 的 tool/confirmation/plan/task/reasoning 對 ai 套件的依賴全部是 import type（type-only、可 inline），不需要 useChat/UIMessage runtime；shadcn base 五件更是純 layout。所以「會不會被迫接上 AI SDK」的答案是不會——但反面是：它們也不提供任何 ACP 資料建模能力，wire → transcript 的 adapter 一行都省不掉。

真正會拖進重相依的是週邊：tool 經 code-block 拖 shiki；reasoning 的 ReasoningContent 綁 streamdown（實測 v2.5.0 連 mermaid@11 都在直接相依裡）、Shimmer 綁 motion。評估建議全部避開。

事實 2 相依疑慮多為假警報：radix-nova 專案裝 Radix 變體即可，Base UI 不會被拖進來

初評把「引入第二套 primitive（@base-ui-components/react）」列為 message/marker 的頭號風險，但反方驗證推翻了這個前提：shadcn 的 base 系列元件都有 Base UI / React Aria / Radix 三種變體，而 Yuzora 的 components.json style 是 radix-nova，CLI 會解析出 Radix 版、重用既有 radix-ui@1.6.2 monopackage——多數元件是零新增相依。同理，src/styles.css 已有完整 shadcn 語意 token layer（light 64-80 行、dark 238-254 行），「元件會整個無樣式」也不成立。

但要立一條全域規則：一律用 radix-ui monopackage 的 Radix 變體，禁止 standalone @radix-ui/react-* 與 @base-ui-components/react——因為 shadcn CLI 產生的 collapsible.tsx 會 import standalone 套件，需手動改回 monopackage import（repo 內 dialog/popover/tabs 全是這個慣例）。

事實 3 真正的瓶頸在 reducer 與 row key，不在 UI 元件

盤點揭露兩個系統性問題，任何元件都解不了、卻是所有「更豐富呈現」的前提：

  • reducer flattening：upsertPlan（acpConnection.ts:1479-1491）把 plan entries join 成單一字串、丟棄 priority；tool status 是不透明字串；BlockEntry.meta 是 JSON 字串。要逐項顯示 status/priority icon，得先改 reducer 停止壓平。
  • 串流 row-key remount：transcript row key 是 `${index}-${entryKey(entry)}`（AgentZonePanel.tsx:1922），而 entryKey 內嵌 entry.text——串流時每個 chunk 都改 key，尾列每個 chunk unmount+remount 一次。這既是現況的隱形效能成本，也直接破壞 message-scroller 的 anchoring/virtualization 與 reasoning 的 duration timer。修 stable-id 是一次性的共同前置。

逐面向確認重點

訊息渲染：不對稱氣泡＋自製 MinimalMarkdown（非 markdown-it）
  • user 右對齊（maxWidth 88%、--yz-active、右下削角 5px）／agent 左對齊（94%、--paper-1、左下削角）；泡泡上方有 uppercase sender 標籤（AgentZonePanel.tsx:1955-2040）。
  • user 訊息是純文字、只有 agent 訊息走 markdown；而該 markdown 是自製 MinimalMarkdown——僅支援 fenced code、inline code、bold，不支援標題/清單/連結/表格。全部輸出 React 節點（無 dangerouslySetInnerHTML），天然免 XSS。repo 裡的 markdown-it＋dompurify 只用在編輯器 MarkdownPreview 與 LSP hover，與 AgentZone 無關——這是盤點點名的最大內容能力缺口。
  • 串流：MsgEntry.streaming＋尾端限定 cursorIndex；chunk 由 reducer 就地串接文字。回合結束 settleTranscript 清 streaming 旗標。
Tool call／permission／plan／thought：ToolBlock＋TranscriptBlock 二分渲染
  • ToolBlock（2067-2101）：手寫 useState(open)＋aria-expanded 可折疊卡片，header 顯示 title＋前 50 字 preview＋status 純字串；展開見 rawInput/rawOutput（JSON.stringify）＋locations（path:line）。status 只有 failed 會轉紅。
  • TranscriptBlock（2112-2210）：其餘 6 kind 共用的彩色卡片（3px accent 條＋meta chip＋action chips）。perm 的選項按鈕就是這裡的 action chips → respondPermission(sessionId, optionId)；ACP 的 N 選項＋四向 kind（allow_once/allow_always/reject_once/reject_always）已正確處理，但答覆後按鈕仍可點（no-op）且 store 不記錄選了哪個——這是現況真實的小缺陷，與是否換元件無關。
  • plan：被 upsertPlan 壓成單一字串（[x]/[wip]/[] 文字標記），displayMeta 顯示 completed/total；永遠全展開。thought：muted 樣式、同樣永遠全展開、無 typing indicator——這兩個「不可折疊」是現況真實 UX 缺口。
捲動與 composer：12 行自製 sticky 捲動；mention／附件已相當完整
  • 捲動（1886-1897）：scrollRef＋80px near-bottom 門檻，串流 chunk instant jump；沒有 scroll-to-bottom 按鈕、沒有 virtualization。
  • composer：auto-grow textarea、send/cancel 切換、IME composition 守衛、/（slash command）＋$（skill）＋@（workspace 檔案 mention）三合一 suggestion popup、圖片附件（貼上＋picker、5MB 上限、讀完驗證才成 chip）；缺 drag-and-drop。附件 chip 是 InputGroup 上的 compact pill，含正確 role=list/listitem 與 testid。
  • usage/cost：header 的 UsageChip（SVG ring＋used/size＋分頁 popover）＋CostChip；infoBanner（startupInfo）用 InfoChip popover 呈現。
測試與樣式體系（影響替換成本的兩個事實）
  • transcript 相關的 data-testid 斷言全部集中在 agentZonePanel.test.tsx 單一檔（約 9 處；appShell/editorPanel 測試並未引用——初評高估 4 倍，已由反方驗證修正）。

bubble SKIP S sound

對應現況

MsgEntry 氣泡本體（1955-2040，反方修正：範圍含 1955-1963 的分支 guard 與 placeholderOnlyText）。radix-nova 變體零新相依、對資料形狀零期望——技術上「能安全採用」。

正因它只是展示殼：現況氣泡已有非對稱削角、sender 標籤、串流游標、圖片縮圖列（比 bubble 多三樣）；bubble 多出的能力（7 variant、BubbleReactions、BubbleGroup）在 ACP 場景全用不到（wire 無 reaction 通道）。要視覺一致得覆寫掉幾乎所有 opinion，只剩 <div data-slot="bubble-content"> 空殼。典型「換皮無淨收益」。

反方驗證

測試面修正為單一檔 5 處；並補註：TranscriptEntryRow 未 memo 的全列重渲問題 bubble 不改善也不惡化，勿誤想「換 bubble 能改效能」。

message-scroller ADAPT M+ sound — 唯一建議引入的套件

對應現況

取代 12 行自製 sticky-to-bottom 捲動（1886-1914）＋entries.map 外殼。headless（@shadcn/react@0.2.1，實測 tarball：零 runtime deps、只 peer react≥19、單一 export），不碰訊息渲染本身。

為何 adapt（淨新增 UX）

  • scroll-to-bottom 按鈕（現況完全沒有）、user-turn scroll anchoring＋previous-item peek、長 transcript 的 content-visibility virtualization、role=log 等 a11y 強化。
  • 反方補充的好消息：virtualization 不是黑盒——[content-visibility:auto] 掛在你可編輯的 MessageScrollerItem className 上，風險可控（串流尾列可單獨拿掉）。

  • 建置地雷：registry 產物 import 了 shadcn 網站內部的 IconPlaceholder（@/app/(create)/components/icon-placeholder），CLI 不會幫你拉——裝完必須手動改成 import { ArrowDown } from 'lucide-react'，否則 typecheck/build 直接紅。
  • stable-id 前置：MessageScrollerItem.messageId 必須跨 chunk 穩定，現況 key 內嵌 text 每 chunk 變動——Phase 0 必做。

  • 捲動行為 jsdom 測不到 → gui-acceptance 實機驗收必跑。

AI Elements 系列

tool SKIP M sound — 改為就地 cherry-pick

對應現況

confirmation SKIP M sound

資料形狀落差（核心）

元件的答覆後顯示 gate 在 approval.approved:boolean（binary），ACP 是任意 options[]＋四向 kind。反方修正：children 可自由渲染所選 option 名稱，所以不是硬性「表達力降級」，但 gate 仍是二元，且無論走元件或自建，都需要同一個 store 改動——respondPermission 目前只刪 pending＋轉 tone，不記錄選了哪個 option（agentStore.ts:1054-1076）。

為何 skip

perm 專屬程式碼其實只有 3 行分支＋共用 action-chip 渲染——可替換面極小。唯一真實 UX 收益（答覆後顯示結果＋disable 失效按鈕）是 store 側改動，與是否採用元件無關，可直接在既有 perm block 補齊。反方另補：shadcn Button 的 :focus-visible ring 與 Alert 的 role=alert 是元件真實的 a11y 優勢，但不足以翻案。

plan＋task 收斂：自建 native PlanBlock M

兩件評估對同一表面互相矛盾 → 由綜整收斂

plan 初評 skip（元件只是可折疊 Card 殼、PlanContent 吃任意 ReactNode、checklist 本體仍要自己刻、shimmer 拖 motion）；task 初評 adapt（目標也是 plan block），但其反方驗證（sound=false）指出：repo 已有 in-idiom、palette-matched、可 jsdom 測的折疊 pattern（ToolBlock 的 useState＋button＋open &&），自建 PlanBlock 在每一個評估軸（相依、樣式一致、測試性）都壓制「adapt task」，而初評沒比較這個支配選項。

收斂結論

不引入任一套件。自建 native PlanBlock：collapsible checklist、[x]/[wip]/[] 換 lucide 狀態 icon、completed/total 續用 displayMeta、沿用 THREAD_KIND_STYLE.plan（--paper-1／accent #5b3fd1）。若要逐項 status/priority icon，前置是改 upsertPlan 停止 flatten（反方修正：壓平發生在 upsertPlan 1479-1491，不是 parsePlanUpdate——後者其實保留了結構）。

reasoning 收斂：自建 native ReasoningBlock M sound

它瞄準的是真實缺口

但 faithful 採用要剝掉的東西太多

  • ReasoningContent 綁 streamdown（反方實測 v2.5.0 直接相依含 marked/rehype 系列＋mermaid@11——比初評想的更重）；Shimmer 綁 motion（只為一個 label）。
  • uncontrolled duration timer 會被現況的 row-key 每 chunk remount 重置——不修 stable-id 就沒有「Thought for Xs」。
  • 硬編英文（i18n 缺口）＋shadcn token 需 reskin。

收斂結論

剝完 streamdown＋motion 後元件只剩 Collapsible 殼＋auto-close timer——自建 ReasoningBlock（抄 ToolBlock idiom＋小 timer＋既有 yzblink/tw-animate-css）拿到同樣 UX、零新相依。反方另修正：monopackage 已 vendor collapsible 與 useControllableState，連 bunx shadcn add collapsible 都不必跑。

建議遷移路線（分階段）

順序是硬約束：三個涉及 transcript 的改動都碰同一個 2428 行大檔與同一組共用 helper，先地基、後替換，否則並行改會在同檔互撞。

Phase 0 — 地基重構（所有後續的共同前置；不採納任何元件）
  • 拆檔：從 AgentZonePanel.tsx 抽出 MessageBubble／PlanBlock／ReasoningBlock／TranscriptBlock，集中共用 helper（THREAD_KIND_STYLE、parseToolBlockMeta、displayMeta、entryKey、dedupeInfoBanner、cursorIndex、TranscriptEntryRow router）。
  • 串流 stable-id：MsgEntry/BlockEntry 加穩定 id（reducer 生成），row key 從 `${index}-${entryKey}`（內嵌 text、每 chunk remount）改為 id。附帶淨賺：修掉現況尾列每 chunk remount 的效能成本。
  • primitive 政策：一律 radix-ui monopackage Radix 變體；禁 @base-ui-components/react 與 standalone @radix-ui/react-*。
  • 集中協調 agentZonePanel.test.tsx 斷言與 en＋zh-TW i18n key（一次做，避免逐件漂移）。
Phase 1 —（可選但強烈建議）reducer 邊界：停止 flatten
  • upsertPlan 保留結構化 entries＋priority；appendToolCall 保留結構化 status。
  • 同步更新 acpConnection reducer 測試與 Session Index replay 相容性。
  • 不做的後果：native block 只能 re-parse 壓平後的文字，UX 天花板受限。
Phase 2 — 自建 native blocks（零新相依）
  • ReasoningBlock：ToolBlock 折疊 idiom＋串流 auto-open／結束 auto-collapse＋i18n「思考中…」。
  • PlanBlock：collapsible checklist＋lucide 狀態 icon＋completed/total chip，沿用 THREAD_KIND_STYLE.plan。
  • ToolBlock cherry-pick：純字串 status → lucide status badge（failed→紅）。
  • （獨立小改）perm 答覆後記錄所選 option＋disable 按鈕——現況真實缺陷，store 側小改。
Phase 3 — message-scroller（唯一引入的套件）
  • bunx shadcn@latest add message-scroller（新增 @shadcn/react@0.2.1）→ 立刻修 IconPlaceholder 壞 import → lucide ArrowDown（否則 build 紅）。
  • messageId 餵 Phase 0 的 stable id；對齊 80px sticky 語意；加 scroll-to-bottom 按鈕；restyle scrollbar/fade 為 .yzs／Yuzora token；aria-label 接 i18n。
  • gui-acceptance 實機驗：逐 chunk 貼底、上捲保留閱讀位置、virtualization 不裁切就地變長的尾列與 ToolBlock 展開。
Phase 4 —（另立 spike）清單外的高槓桿元件
  • agent 訊息 markdown 升級：AI Elements Response（Streamdown）vs 擴充既有 markdown-it＋dompurify 管線——傾向後者（零相依 delta、避開串流 hot path 的 async re-highlight）。
  • Prompt Input＋Suggestion：composer 是 AgentZone 最大未評表面（缺 drag-and-drop 等）。
  • Context：對應 UsageChip/CostChip，多出 input/output/reasoning/cached token 分項，屬能力升級。

開放問題（需要你拍板）

  1. scroll-to-bottom 按鈕＋長 transcript virtualization 是否為真實痛點？若否，message-scroller 也應改 skip（現況 12 行捲動可用），本輪就變成純地基重構＋native block。
  2. 是否把整個 transcript 遷往單一 shadcn token 樣式體系？只有在此前提下 tool/confirmation/plan/task/bubble 才可能翻成 adopt；否則維持 skip 以避免孤島。
  3. Phase 1（reducer 停止 flatten）做不做？這是結構化 plan/tool 呈現的真正 gate，但動到 acpConnection 測試與 Session Index 持久化相容性。

  5. 清單外的 Prompt Input／Suggestion／Context 是否納入範圍？它們不在原始請求內，但對 AgentZone 的槓桿比本次 10 件中的大多數更高。
