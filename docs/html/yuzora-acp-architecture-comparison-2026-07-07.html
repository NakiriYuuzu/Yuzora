Agent 整合架構評估：Zed ACP vs cmux vs tempo-term

三個開源專案的 coding-agent 整合方式實地比對（各自 source 逐檔查證），評估哪種路線最適合 Yuzora 的 AgentZone。

結論（TL;DR）

維持現行方案 A′（Zed 式 ACP，前端 host）。三者中只有 Zed 與 Yuzora 同屬「協定型」整合，而 Yuzora 的 M5 設計本來就是以 Zed 為藍本裁剪到 Tauri 雙層架構的——這次逐檔比對確認該裁剪與 Zed 實際實作方向一致（含 UI-thread 回呼、stderr tail、exit 競速、trace log 等細節）。

cmux 與 tempo-term 走的是觀測型路線：agent 以原生 TUI 跑在 terminal 裡，宿主靠 per-agent hooks／transcript 檔案 tail 取得狀態。這條路換不到 AgentZone 需要的結構化 transcript、in-app 核可與 diff preview——cmux 自己的設計文件就承認，要把 streaming 弄進 GUI 只剩「螢幕擷取＋逐 agent 去雜訊」這種先天啟發式的做法。

可帶走的東西：Zed 的 configOptions／model picker（pi-acp 已回傳該欄位，Yuzora 目前丟棄）、non-interactive shell spawn（Zed 從構造上避免 login-shell 噪音）、cmux 式桌面通知（agent 等核可且視窗未聚焦時）；tempo-term 的 hook→OSC 招式則是未來 terminal drawer 想「觀測手動跑的 CLI agent」時的現成低成本配方（同 Tauri stack）。

Yuzora 現況（比較基準）

M5 已交付 AgentZone ACP client（方案 A′，commit 4d0e2f0）：

  • 前端以官方 TS SDK @zed-industries/agent-client-protocol 的 ClientSideConnection host 協定（src/agent/acpConnection.ts）；Rust 端 agent_process.rs 只做啞管線（spawn／stdin 寫入／stdout 逐行 event／stderr 環形緩衝／JSON gate／PATH preflight）。
  • fs 回呼走 editor buffer 層（dirty buffer 讀取、save 路徑寫入）；permission 依 ACP option kinds，allow_always 記 session 記憶體；ACP terminal 能力橋接 agent_terminal_*；auth 走 terminal login；loadSession resume；trace toggle → JSONL Logs。
  • First-class agent 為 pi（經 bunx pi-acp adapter），Settings 可自訂任何 ACP agent command。

設計時的業界調查（design doc §2.3）只看過 Zed／Superconductor／Orca；cmux 與 tempo-term 是本次新增的比對對象。

三種做法

先講最重要的分野：協定型（宿主與 agent 之間有結構化雙向協定，agent 沒有自己的畫面）vs 觀測型（agent 的 TUI 就是畫面，宿主從旁觀測狀態）。Zed 屬前者；cmux、tempo-term 屬後者。這個分野決定了你能在自家 UI 裡做到什麼。

A · zed-industries/zed

協定型：ACP 原生 client

協定型（ACP 作者） native Rust / GPUI GPL-3.0（SDK 為 Apache-2.0）

Zed 是 ACP 的制定者與參考實作。agent 以子行程跑 stdio JSON-RPC，完全沒有 TUI；transcript、tool call、diff、permission 全部是結構化資料，由 Zed 原生渲染成 agent panel。

機制

  1. AgentServer trait：內建 gemini／claude-acp／codex-acp／cursor 四個 preset＋settings 定義的 CustomAgentServer。
  2. AcpConnection::stdio：以 non-interactive shell spawn（ShellBuilder::non_interactive()），env 顯式傳入——從構造上杜絕 login-shell profile 噪音。
  3. 逐行 transport 前後掛 debug tap：進／出／stderr 全記到 2000 筆環形緩衝（AcpDebugLog），供 acp_tools 的即時協定檢視器。
  4. 連線期間把 child exit status 與 handshake／initialize 競速——agent 死掉時回 LoadError::Exited 附 stderr tail，而不是等 timeout。
  5. client 回呼經 foreground dispatch channel 全部送回 UI thread（GPUI entity 是 !Send）——「ACP client 回呼本質是 UI 事務」在 Zed 是實作出來的事實。
  6. acp_thread crate：connection-agnostic 的 AgentConnection trait＋AcpThread session 模型＋ActionLog（buffer 層編輯追蹤，可整批 review／reject）。

Pros

  • 結構化 streaming／tool call／diff／perm，UI 想怎麼畫就怎麼畫
  • agent 廣度靠生態系 adapter，宿主不用逐家維護
  • 錯誤面處理最成熟（exit 競速、版本門控、auth 分流）
  • 與 editor buffer 深度整合（dirty buffer、edit review）

Cons

  • 依賴 adapter 保真度（協定外行為看不到）
  • 放棄各 agent 自家 TUI 的完整體驗
  • 單進程 native 架構，不能整包照抄到 Tauri

B · manaflow-ai/cmux

觀測型：terminal 宿主＋hook 生態

觀測型 native Swift / AppKit＋libghostty GPL（dual-license）

cmux 是「給 agent 用的終端機」：agent 以原生 CLI TUI 跑在 ghostty pane 裡，cmux 完全不碰協定，改以 per-agent hooks＋環境變數 token 從旁建立可靠的狀態通道。

機制

  1. 每個 terminal spawn 時注入 CMUX_SURFACE_ID 等 env token；hook 觸發時以「flag → env → tty 對照 → process tree」四層解析回綁 surface（明文禁止 title／mtime 啟發式）。
  2. cmux hooks setup 對 16 種 agent 各寫一份 hook／extension 到該 agent 自己的設定檔（Claude 走 wrapper 注入 --settings、pi 走 extension…），hook 經 socket 回報 app。
  3. 換得：lifecycle badge（running／idle／needsInput）、通知、Feed 核可卡（橋 PermissionRequest）、以各家原生 resume 指令復原 session、閒置 agent hibernation。
  4. iOS GUI 模式讀 agent 自己的 transcript JSONL 重建對話；GUI streaming 只能靠讀 ghostty 螢幕格網＋逐 agent 剝 TUI chrome，其設計文件自評為「large, inherently heuristic subsystem」。

Pros

  • agent TUI 體驗零損耗，任何 CLI 首日可用
  • 多 agent 併行管理的工程（通知、hibernation、resume）非常完整
  • token 綁定設計嚴謹，值得任何「觀測」需求借鏡

Cons

  • 16 份 hook adapter 全數自維護，隨各家 CLI 改版漂移
  • 桌面端沒有結構化 transcript——terminal 就是 UI
  • in-app 核可／diff preview 做不到協定級的精確
  • GUI streaming 需螢幕擷取，作者自認先天啟發式
證據（本次讀過的檔案）
  • docs/agent-hooks.md — 16 agent 整合表、~/.cmuxterm/<agent>-hook-sessions.json、hibernation、PushNotification 橋
  • docs/agent-session-tracking-spec.md — env token 綁定原則、四層 surface 解析、「no heuristics」宣言
  • docs/streaming-agent-updates.md — JSONL 無 token 級增量的實證；ghostty 螢幕擷取是唯一可行來源的結論
  • docs/notifications.md、Sources/Agent*.swift（結構掃描）
C · mukiwu/tempo-term

觀測型輕量：hook→OSC＋transcript tail

觀測型 Tauri 2＋React 19（與 Yuzora 同 stack） Apache-2.0

AI-native terminal workspace（terminal＋editor＋git＋SSH＋BYOK 助理）。agent 一樣跑原生 TUI；獨到之處是狀態通道不走 socket，而是讓 hook 直接把自訂 OSC 序列寫回 agent 的 tty，由前端 xterm.js 解析。

機制

  1. 把 hook script 合併進 ~/.claude/settings.json（idempotent 純函式，Codex 另有一套）；事件對映：UserPromptSubmit→thinking、PreToolUse→active、PermissionRequest→waiting-approval、Stop→idle…
  2. hook 無控制終端，於是沿 process ancestry 找到 agent 的 tty，把 OSC 6973 直接寫進 /dev/<tty>；前端 xterm 解析後變成 tab 卡片上的狀態 badge＋未聚焦時桌面通知。以 TEMPOTERM=1 env 做宿主 gate。
  3. Rust claude_progress／codex_progress 以 notify watcher tail agent 自己的 transcript JSONL → session 卡片與自動標題。
  4. 「AI assistant」是完全獨立的 BYOK chat 模組（直連 OpenAI 相容／Anthropic／Ollama endpoint），與 terminal 裡的 agent 無關。

Pros

  • 與 Yuzora 同 stack，任何招式可直接平移
  • OSC 通道極輕：無 socket、無 daemon，順著 PTY 就回來了
  • hook merge 寫成純函式可測，工程品質好

Cons

  • 只支援 Claude Code＋Codex 兩家，逐家硬編
  • 純觀測、唯讀：badge＋通知而已，無 in-app 核可
  • 無結構化 transcript／tool call／diff 流
  • 寫 /dev/tty 在沙箱／權限收緊環境有脆性
證據（本次讀過的檔案）
  • src-tauri/src/modules/claude_status_hook/status-hook.sh — process-ancestry 找 tty、printf '\033]6973;…\007' > /dev/$tty
  • claude_status_hook/mod.rs — EVENTS 事件表、settings.json 合併純函式
  • claude_progress/mod.rs — transcript JSONL tail＋標題萃取
  • modules/ai/provider.rs — BYOK chat（含 loopback-only http 白名單）

兩條路線的資料流。關鍵差異：協定型的資料通道是「雙向、結構化、逐 token」；觀測型是「單向、事件級」，內容資料要另外從 agent 的 transcript 檔或螢幕反推。

哪個適合 Yuzora？

建議：維持 Zed 式 ACP（方案 A′），觀測型僅作未來 terminal drawer 的補充層

1. 產品形態決定路線。cmux 與 tempo-term 的產品是「terminal 優先，agent 是房客」——TUI 本身就是 UI，觀測型只需要補 badge 和通知，成本效益極高。Yuzora 的 AgentZone 是「workbench 優先，agent 是一等面板」：transcript 氣泡、tool block、diff preview/apply、permission 按鈕都是 MVP spec 明定的結構化 UI，這些資料只有協定能供給。

2. 觀測型換不到 Yuzora 已經做到的事。最有力的證據來自 cmux 自己：其 streaming 設計文件實測 Claude Code／Codex 的 JSONL「整塊落地、無 token 增量」，結論是把 streaming 弄進 GUI 唯一可行的來源是 ghostty 螢幕格網擷取＋逐 agent 剝 TUI chrome，並自評這是「大型、先天啟發式、難以確定性測試」的子系統。Yuzora 走 ACP 之後這些是免費的。

3. 本次比對驗證了 A′ 的裁剪方向。Zed 的 client 回呼實際上全部經 dispatch channel 送回 UI thread（acp.rs:883）——這正是 A′「回呼本質是 UI 事務、應住 webview」的立論；Yuzora 的 exit 事件＋stderr tail＋initialize timeout 組合，等價於 Zed 的 exit-status 競速；trace→Logs 對應 Zed 的 AcpDebugLog。差距只剩下面「可借鑑」清單裡的增量項。

4. 觀測型的正確位置。Yuzora 已有 terminal drawer；使用者遲早會在裡面手動跑 claude／codex。屆時 tempo-term 的 hook→OSC 配方（同 Tauri stack、無 socket）是給這些「drawer 房客」補狀態 badge 的最低成本做法——這是 AgentZone 的互補功能，不是替代。列 v2 選項即可，現在不動。

風險對照（兩條路線各自的坑）

ACP 路線：adapter 保真度已發生

協定型的能力上限＝adapter 的品質。Yuzora 一手證據：pi-acp 0.0.x 在內層 pi 行程死亡時 unhandled EPIPE 整個 adapter crash，client 端只看得到行程消失（docs/upstream-issues/pi-acp-epipe-2026-07-07.md）。緩解已到位：exit event＋stderr tail＋EPIPE 判別訊息＋Logs 指引；長期靠上游修復與多 agent 分散。

ACP 路線：協定外行為看不到中

M5 spike 實測 pi 自己落盤寫檔、不走 fs/write_text_file 也不發 permission——「核可後才寫入」的保證取決於 agent 端配合。觀測型同樣有此問題（hook 也攔不住 agent 自己動手），並非換路線可解；靠 watcher 把繞過的寫入視為外部變更已是正確兜底。

觀測型路線：heuristic 漂移高（若採用）

cmux 文件自證：TUI chrome 隨 CLI 版本與 locale 漂移、需要 recorded byte fixtures 才能測、16 份 adapter 逐家維護。tempo-term 較輕但同構（2 家硬編、/dev/tty 寫入在權限收緊環境脆弱）。這是 Yuzora 設計文件當初把 Orca 列為反面教材的同一類成本。

共同：生態尚年輕低

ACP adapter 多為 0.x（pi-acp 0.0.31）；hook API 也各家自訂、無標準。兩條路都押在快速演化的生態上——差別在 ACP 有 Zed／Google 背書的標準化協定與 schema 版本門控，漂移成本由協定吸收，觀測型由宿主吸收。
