#16 [Windows] Terminal 啟動後不在目前專案資料夾 P1 ready-for-human working tree 已修 effort S
根因

workspace 路徑經 std::fs::canonicalize（fs_service.rs:39-45）後帶 Windows verbatim \\?\ 前綴，一路存進 workspaceStore、傳到 pty_open（pty_service.rs:148-152）直接當 shell cwd。cmd.exe 視為 UNC 印出「UNC paths are not supported, Defaulting to Windows directory」切到 C:\Windows；PowerShell 落到 profile home。issue 原推測的「shell profile 再次切目錄」並非主因。

Working tree 狀態

已含針對性修法：pty_service.rs 新增 shell_spawn_cwd()（53-76 行，strip drive 前綴、UNC 轉回雙反斜線、其他 namespace 原樣保留），並附 3 個平台無關單測＋2 個 #[cfg(windows)] 整合測試（fixture 含空白／中文／#／% 的「專案 空間 #100%」）。判 likely-fixed 而非確定，因 Windows 端測試不在 CI 執行。

建議

維持治標方向（治本牽動 canonical key 索引，見分群 2）。Windows 實機以 gui-acceptance 驗 PowerShell／cmd／自訂 shell 皆從 workspace root 啟動後，經使用者授權拆成獨立 commit。建議讓 ci.yml 的 Rust tests 也在 windows-latest 跑，否則此修復無自動守門。

風險與待決

Volume GUID 等罕見 namespace 仍可能被 shell 拒絕；PtySessionInfo.workspace 仍存 verbatim 原值（僅 metadata）。待決：是否升級為 fs_service 治本以同時解 #17（覆核與 #17 分析都建議：不要，維持治標）。

#11 [Windows] LSP 一鍵安裝失敗 P1 ready-for-agent 未處理 effort M
根因（強預測，issue 未附 log）

npm 路由（vtsls＝TS 預設、pyright＝Python 預設、typescript-language-server）必失敗：lsp_download.rs:839 用 which("npm")，而 Windows 候選名把裸名 npm 排最前（framing.rs:142-152）、可執行判定只是 is_file()（framing.rs:186-189）——Node.js 安裝目錄同時放了無副檔名的 npm（Unix sh 腳本）與 npm.cmd，於是解析到非 PE 的 sh shim，Command::new(裸npm).spawn() 以 ERROR_BAD_EXE_FORMAT 失敗。正確範式（cmd.exe /C 包裝）已存在於 process_service.rs::shell_command，只是沒被 LSP 路徑重用。binary 路由（rust-analyzer 等，純下載寫檔）應可成功；pip 路由風險次要（Windows Store python alias stub）。

建議修法
  • execute_plan 的 Npm 分支在 Windows 以 ComSpec＋/C 包裝，注意 CVE-2024-24576 參數轉義（raw_arg 或嚴謹引號），保留 timeout／kill_tree／去敏 stderr。
  • 修 candidate_names_for：Windows 優先 .cmd/.exe 而非裸名。
  • 命令組裝是吃 windows: bool 的純函式，可在 macOS/Linux CI 直接單測守門，不需 Windows 機器。
  • 次要強化：detect_python 增列 py -3、過濾 WindowsApps Store alias。
覆核勘誤

覆核者指出「CreateProcessW 無法執行 .cmd」的括號主張不精確（Rust 給完整路徑其實可 spawn 批次檔，CVE-2024-24576 正因此存在），但標準安裝下 which 根本輪不到 npm.cmd、主根因不受影響。clippy 是 exact baseline，改動後記得同步 .github/clippy-baseline.json。

#21 [Windows] Alt+F4 關閉未提示未儲存變更 P1 ready-for-agent 未處理 effort M
根因

AppShell.tsx:207-230 的 onCloseRequested 只做 pty／dev-server 清理，從不呼叫 event.preventDefault()、不檢查 dirty——且這是現行刻意設計（appShell.test.tsx:450-472 明確斷言 preventDefault 未被呼叫）。Rust 端 RunEvent 無法顯示 React 模態，攔截必須在 JS 層。mixed-EOL 被 blocked 的文件（saveDocument.ts 的 {kind:"blocked"}）在 Alt+F4 後直接消失。全 codebase 無任何 draft／autosave 機制。

建議修法

從 workspaceActions 抽出 collectDirtyPaths()；onCloseRequested 改 async：preventDefault → 無 dirty 直接清理＋destroy()；有 dirty 走 requestUnsavedDecision（cancel 不關；save 逐一 saveDirtyTab，任一非 saved 即中止退出；discard 或全存成功才 destroy）。改寫既有測試為四情境；補 en/zh-TW 文案。draft recovery 另立 spec，本 issue 先交付 close-guard。

風險

攔截處理不當會讓視窗關不掉（所有分支的最終路徑要明確）；destroy() 迴避遞迴觸發；需確認 macOS ⌘Q 是否也走 onCloseRequested（acceptance 要求一致）。

#22 系統佔用只統計 parent process P2 ready-for-agent 未處理 effort M
根因

perf_service.rs:26-37 的 sample() 只刷新自身 PID（ProcessesToUpdate::Some(&[pid])），從不走訪 process tree。Terminal shell、ACP node wrapper、Pi agent、dev server 全是 Yuzora 的 descendants（各 spawn 路徑已逐一確認），卻不在統計內——macOS 實測 parent 約 102MiB vs 全樹約 370MiB。i18n 文案甚至直白寫「主程序 CPU · 記憶體」。

建議修法

sample 改 ProcessesToUpdate::All＋parent→children 鄰接表 BFS 去重聚合（進階替代：以 PtyManager／AgentManager／ProcessManager 掌握的 pid 為種子做 subtree 聚合，較精確但接線成本高）。CPU 語意需拍板：維持「每核 100% 加總可 >100%」或除以核心數正規化——影響 StatusBar 呈現與 i18n。測試用 sh -c 'sleep 30 & sleep 30 & wait' 造祖孫 fixture。

風險

每 2s 全量列舉行程的成本（有失焦跳過節流可緩解）；新 descendant 首輪 CPU 讀 0 屬預期；re-parenting 邊界（中間層死亡→孫行程脫離統計）建議註明為已知限制。此項同時是 #9 精準量測的前置。
