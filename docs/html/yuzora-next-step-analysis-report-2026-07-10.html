Project Analysis · Yuzora 0.1.0

現在不該再加功能；下一步是把「已做完」變成「可證明可交付」

一句話判斷

專案不是「功能不足」，而是「完成狀態沒有被同一套證據閉環」。

已具備

核心功能與自動測試

M1–M6、ACP、DB/SSH、logging、preview、terminal 等主體已落地；目前 build、1036 tests、cargo check、clippy、298 Rust tests 均通過。

尚未閉環

GUI / 原生 / 外部系統

M5 記錄仍是 GUI 0/32；M6 與 ACP checklist 目前未勾。child webview、auth、真 DB/TLS、SSH/SFTP atomic、dirty modal、Windows 行為都需要實機證據。

真相分裂

文件與現況不一致

M4 原始決策是 iframe-only，現況卻是 local iframe + external child webview；ACP 結案報告同時寫 waived、未完成與舊 build 紅燈，已不適合作為唯一真相源。

重要：不要直接照舊 checklist 從頭跑。先建立一份「以目前程式碼為準」的統一驗收矩陣，將 M4 iframe-only 等過時項標記 superseded，否則會驗到不存在的舊架構。

從 MVP 到 RC1 的位置

原始 roadmap 到 M5 結束；M6 是後續回饋批次（其計畫文件仍標示草案），ACP 是再後續的架構整合，目前沒有正式 M7。

判讀：功能曲線已走到後段，但證據曲線落後。RC1 的目的不是再造一套功能，而是讓兩條曲線重新對齊。

最先做的五件事

推薦執行順序

每一階段都有明確 exit gate；前一階段沒過，就不進下一階段。

P0

建立可相信的自動基線

只處理會影響本批次交付的 gate，不擴張成全 repo 大重構。

  • cargo fmt --check：目前在 db_service.rs、env_path.rs、ssh_service.rs 失敗。
  • Changed-file ESLint：3 warnings，位於 AgentZonePanel、LogsSection、AgentBridge。
  • 重跑 lint / typecheck / test / build / fmt / clippy / cargo test。
Exit gate
完整 CI parity exit 0；不以「build/test/cargo check 三件套」取代 CI。
P1

重寫一份目前有效的統一驗收矩陣

舊 checklist 先做 truth reconciliation，再決定哪些要執行。

  • M4 iframe-only 項標記 superseded，改驗 hybrid preview。
  • ACP closeout 中的 waived / 未完成 / build 紅燈矛盾改由目前證據覆蓋。
  • 每項標註 platform、外部依賴、data-loss / security / native-layer 等級。
Exit gate
所有未驗項都能追到當前程式碼與明確驗證方式。
P2

macOS 高風險實機驗收

先驗最可能產生不可逆後果或整合失敗的路徑。

  • Preview child webview：resize、overlay、askpass、SymbolPicker、close/unmount。
  • ACP：pi / Claude / Codex picker、auth terminal、workspace session retention、multi-agent。
  • DB/SSH：PG TLS / trustCert、MSSQL、SFTP atomic、known_hosts fail-closed。
  • Dirty modal、workspace restore race、log secret redaction、opener scope。
Exit gate
高風險項逐條 PASS/FAIL，有 screenshot / log / fixture 證據。
P3

Windows 專項與交叉平台確認

macOS PASS 不代表 multiwebview、path、ConPTY 或 Job Objects 在 Windows 成立。

  • child webview、canonical \\?\C:\…、ConPTY、Job Objects。
  • MSSQL 實機、SFTP drag/drop、工作管理員效能數值。
  • 若無 Windows 環境，明確把 RC1 標為 platform-limited，不偽稱跨平台完成。
Exit gate
Windows PASS，或有被接受且文件化的 release limitation。
P4

收斂 baseline 並設計 M7

到這一步才決定下一輪產品投資。

  • 唯一 closeout + 唯一 current acceptance checklist。
  • Preview hybrid 寫 ADR：正式接受 unstable child webview，或回退簡化。
  • 建立 release artifact 流程；updater、全面 lint 清債先維持非目標。
Exit gate
RC1 有版本、證據、限制與可重現驗證命令。

建議的 M7 定義

Proposed Milestone

Goal

  • 把 Yuzora 0.1.0 轉成有證據的 RC。

架構判讀：哪裡該深、哪裡先別動

保留既有深 module

db_service.rs、ssh_service.rs、logging.rs 雖然都超過 1,000 行，但它們把多引擎、連線生命週期與檔案格式藏在集中 interface 後面。檔案長度本身不是拆分理由。

原則：下一步只在驗收發現 caller 必須知道太多內部規則時，才調整 seam。

Agent seam 需要觀察

目前行為分散於 agentRouter、acpConnection、agentStore、AgentBridge 與 UI。Router 已把 provider/cwd routing 隱藏起來，是值得保留的 seam。

原則：先完成 real-agent acceptance；若 provider-specific 分支持續外漏，再把 runtime/auth deeper 到單一 interface，別在驗收前重構。

Preview 是目前最明顯的 seam leakage

child webview 是 native layer，必須讓 CommandPalette、Settings、ContextMenu、DiffModal、SymbolPicker、askpass 都協調 hide/show。這代表 preview implementation 的 z-order 規則已外漏到多個 caller。

決策：要嘛以 ADR + acceptance 正式接受這個成本；要嘛回到較小 interface（local iframe + external open）。

大型 UI hotspot 不宜趁機全面拆

AgentZonePanel.tsx 1,398 行、acpConnection.ts 1,177 行、agentStore.ts 751 行。這些是未來變更風險訊號，但 RC1 只應處理與 current gate / acceptance 直接相關的部分。

刪除測試：若抽出 module 後，刪掉它只會讓同樣複雜度散回多個 caller，才算真正有 depth。

已知矛盾與需要裁決的點

Preview：M4 iframe NO-GO child webview vs 現況 hybrid
M4 spike 明確判定 unstable child webview 為 NO-GO；現況 Cargo 已啟用 tauri/unstable，外部 https 使用 child webview，local server 留 iframe。這可能是合理的 M6 後續決策，但缺少一份 superseding ADR 與跨平台實證。
ACP：closeout 宣稱 waived，checklist 與 open section仍顯示未完成
目前應以 2026-07-10 實測（1,036 tests / build / cargo check）取代舊 build 紅燈；但 real-agent GUI 仍沒有證據，因此不能只靠「waived」推定功能完成。
CI：quick facts 三件套 vs GitHub Actions 完整 gate
專案日常 quick checks 是 build/test/cargo check；正式 CI 還包含 frontend lint/typecheck 與 Rust fmt/clippy/test。此次正好證明 quick checks 全綠仍可能讓 CI 因 rustfmt 失敗。
Roadmap：MVP 到 M5，M6/ACP 為後續批次，沒有正式 M7
因此「下一步」不是執行某個現成 M7，而是先完成 RC1 收斂，再用當時的實證決定 Agent-first、Remote/Data-first 或 Release-first 的正式 spec。
