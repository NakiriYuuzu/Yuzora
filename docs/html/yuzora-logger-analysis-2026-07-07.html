yuzora Logger 現況問題分析

Claude Code · 2026-07-07 · 分析對象：src-tauri/src/logging.rs、git/ssh/process_service 寫入端、SettingsDialog LogsSection、~/.yuzora/logs 實際資料

TL;DR

Logger 骨架（JSONL 日切檔、query/export/sources 三命令、sanitize 匯出）運作正常，但有 3 個高優先問題：① cargo test 會把測試用 git 指令灌進正式 log（07-06 當日 29,919 筆中 29,674 筆是 run_git）；② log_query／log_sources／log_export 是同步 command，在 main thread 整檔讀入最多 100MB，且 Logs pane 的文字篩選每個 keystroke 都觸發查詢、無 debounce；③ log_sources 用 limit: usize::MAX 全量解析只為取 distinct sources。

ACP 診斷（2026-07-07 追加）：log 中 19 次 agent spawn 沒有一次成功建立可用連線，呈三階段故障——① 8 次 exit 127（zsh:1: command not found: bunx，打包 app PATH 斷）；② pi-acp 版本問題（舊版不回應 initialize；19:16 新下載版回應了但 18 秒後 unhandled EPIPE 整個 crash）；③ 其後全部「零輸出 60 秒 → initialize timeout kill」。CLI 以完全相同指令實測 1 秒內回應，app 內卻全靜默——而 acp trace 預設關閉、spawn 不記 command、kill 不記原因，現有 log 不足以定位第三種失敗，這本身就是 logger 對 ACP 的最大缺口。修復計劃見「修復計劃」tab。

結論：正式 log 幾乎被 git_service 的 debug 事件淹沒，其中絕大多數來自 cargo test 而非真實使用。

P0 — 高優先

P0-1cargo test 直接汙染正式 log 目錄（單日近 3 萬筆）

src-tauri/src/git_service.rs:144 的 log_git_call 無條件寫入 logging::default_log_dir()（即 ~/.yuzora/logs），log 目錄沒有任何注入點。git_service 的單元測試每跑一次 run_git 就寫一筆——07-06 當日 git init -b main×2,900、git config user.name t×2,900、git commit -m c1×2,096⋯⋯全部進了正式 log。

  • 後果一：正式 log 被灌爆，Logs pane 的 debug kind 幾乎不可用；retention 100MB 上限會被測試加速吃掉，反而把「真的」舊 log 提早刪除。
  • 後果二：export bundle 給人 debug 時，附帶大量無意義測試雜訊。
  • ssh_service.rs:103、process_service.rs:69 同樣在 production 建構子裡直接 LogSink::new(default_log_dir())，測試若走到該路徑也會外洩（process_service 測試有注入 Vec<LogEvent>，git_service 沒有）。

建議：把 log dir（或 LogFn）注入 run_git 所屬結構，比照 process_service 的 LogFn 模式；至少在 #[cfg(test)] 下改寫到 tempdir。

P0-2log_query／log_sources／log_export 為同步 command，main thread 做重 I/O；前端無 debounce

logging.rs:192,209,298 三個 command 都是同步 fn。Tauri v2 同步 command 在 main thread 執行，而 query_dir（logging.rs:173）用 read_to_string 一次把整個檔案載入（實測單檔已達 6.7MB，14 天保留 + 100MB 上限下最壞可近 100MB）。文字篩選沒命中時會把全部檔案掃完。

前端放大了這個問題：SettingsDialog.tsx:697-715 的 effect 依賴 text，每個 keystroke 立即 invoke 一次 log_query，沒有 debounce 也沒有取消機制（alive flag 只丟棄結果，查詢本身照跑）。連打幾個字 = 連續多次全檔掃描。

建議：三個 command 改 async fn +（必要時）spawn_blocking；query_dir 改 BufReader 逐行；前端 text/since/until 加 250–300ms debounce。

P0-3log_sources 以 limit=usize::MAX 全量解析，只為取 distinct sources

logging.rs:196-206：sources_dir 呼叫 query_dir 且 limit: usize::MAX，把 14 天內所有 record（含 metadata JSON）解析成 Vec<LogRecord> 塞進記憶體，再取 source 欄位。以目前 7 萬筆資料就是一次 7 萬個 LogRecord 的配置；Settings Logs pane 每次打開都執行（SettingsDialog.tsx:679-691）。

建議：逐行 streaming、只 parse 出 source 欄位（或用 serde_json::from_str::<Value> 取單鍵）直接塞 BTreeSet，不要累積 record。

P1 — 中優先

P1-1多寫者各自建 LogSink 繞過 managed state；torn line 風險由 query 端靜默吞掉

lib.rs:36 manage 的 Mutex<LogSink> 只服務前端 log_event；git_service（每次呼叫新建一個 sink，含 create_dir_all）、ssh_service、process_service 各自持有獨立 sink，同時 append 同一個檔案。writeln! 走 write_fmt，不保證單一 write syscall，多執行緒併發下有交錯（torn line）可能。而 query_dir 對 parse 失敗的行靜默跳過（logging.rs:177），壞行永遠不會被發現。

建議：集中成單一共享 sink（Arc<Mutex<LogSink>> 或 mpsc channel + 單一 writer thread）；write 先組完整含 \n 的 String 再單次 write_all。

P1-2timestamp 用本地時區 RFC3339 + 字典序比較，時區改變即失效

寫入端用 chrono::Local::now().to_rfc3339()（實際值如 2026-07-07T00:36:21.853757+08:00），查詢端 logging.rs:148-157 直接做字串 >=／<=。同一 offset 下字典序恰好正確，但只要 offset 改變（出國、DST、或與帶 Z 的外部紀錄混雜）比較就錯；since/until 也沒有 parse 驗證，亂輸入等於靜默全過或全擋。

建議：寫入改 Utc::now().to_rfc3339()（UTC 字典序 = 時間序），查詢端 parse 成 DateTime 比較；顯示層再轉本地時區。

P1-3cleanup 只在啟動跑一次；size 超限清理會誤刪目錄內任何檔案

lib.rs:30-31 只在 app 啟動時呼叫一次 cleanup()；長時間不重啟則 14 天 retention 與 100MB 上限都不會執行。且 logging.rs:76-108 的 size 清理把 read_dir 掃到的所有檔案（含 legacy 的 yuzora-logs.sqlite*、甚至今天正在寫的檔）都納入刪除候選。

建議：size 清理只針對符合 yuzora-YYYY-MM-DD.jsonl 模式的檔案，並排除今日檔；另在日切換或定時（例如每次 rotate 時）補跑 cleanup。

P1-4sanitize 只遮 home path 與 username，credentials 會原樣進 log 與 export

log_git_call 把完整 git args 寫進 message（已避開 extra_env，good），但像 git remote add origin https://user:token@host/repo 這種帶 credentials 的 URL 會原樣入庫；export 的 sanitize_line（logging.rs:230-242）只處理 home 與 /Users/<user>/，不遮 token。另外 Logs pane 的 text 搜尋只比對 event／message（logging.rs:158-162），git 錯誤內容存在 metadata.stderr，搜不到。

建議：寫入端對 args 做 URL userinfo 遮罩；text filter 加上 metadata 序列化字串比對（或至少 stderr 欄位）。

P2 — 低優先

P2-1舊制殘檔無人管理

~/.yuzora/logs 內殘留 yuzora-logs.sqlite／-shm／-wal（約 380KB）與 yuzora-logs-20260701-180447.848-0.jsonl。query／export／age-cleanup 都不認得它們，只有 size 超限時可能被「誤刪」。建議一次性遷移或刪除，並在 cleanup 明確處理。

P2-2logUserAction 無 .catch，失敗成 unhandled rejection

src/features/logs/userAction.ts:8-18 回傳 promise 但呼叫端一律 void logUserAction(...)；backend 失敗（如 Mutex poisoned）會變 console unhandled rejection。建議在 userAction.ts 內部 .catch(() => {}) 吞掉——log 失敗不該打擾使用者。

P2-3寫入錯誤全靜默

LogSink::write 的序列化 unwrap_or_default、開檔與 writeln! 全部 let _／if let Ok 忽略。磁碟滿或權限問題時 log 無聲消失。可接受的設計取捨，但至少序列化失敗（寫出空字串行）應該跳過而非寫入空行。

P2-4retention 邊界實為 15 天

logging.rs:88 用 > RETENTION_DAYS，含今日實際保留 15 個檔。與「14 天」語意差一天，nit。

P2-5log_event 接受任意 level/kind/source；UI 篩選是固定清單

前端 LOG_KIND_OPTIONS = ["debug","user_action","audit"]、levels 四值固定（SettingsDialog.tsx:614-615），但 log_event 不驗證欄位值。未來若有人寫入清單外的 kind/level，該紀錄永遠篩不出來（只能靠「全不選 = 不過濾」看到）。建議後端以 enum 收斂或前端動態聚合。

結論：19 次 spawn、0 次可用連線，三階段故障

07-06 11:13 ～ 07-07 00:37 全部 19 次 acp_exit 的時間軸

結論：故障型態隨時間演變——PATH 斷 → pi-acp 版本問題 → 全靜默 timeout；最後一種佔據所有近期嘗試。

三階段故障的證據與判定

階段①exit 127 ×8 — 打包 app 的 PATH 沒有 bunx

8 次 exit 都留下唯一一行 stderr：zsh:1: command not found: bunx，cwd 為 fixtures/out（gui-acceptance 實機驗收環境）。代表當時執行的打包 app 中 env_path::fix_gui_path() 沒生效或該 build 尚未包含修正。

關鍵缺口：env_path.rs 的成敗只走 eprintln!（stderr），完全不進 log 系統——從 log 無法回答「那時 PATH 修正到底跑了沒、抓到什麼」。

階段②pi-acp 版本問題 — 舊版不回應、新版 unhandled EPIPE crash

19:10／19:15 兩次（agent-9/10）：inner pi 有啟動（stderr 出現 GEOSENSE_API_KEY／YUUZU_API_KEY is missing、No models match "yuuzu-router/gpt-5.5" 等 provider 警告），但 stdoutSeen:false——當時快取的 pi-acp 版本從未回應 initialize，7–9 秒後被 kill。

19:16（agent-1）：bunx 現場下載新版 pi-acp（Resolved, downloaded and extracted [6]），這次 stdoutSeen:true（有回應！），但 18 秒後 pi-acp 在 _PiRpcProcess.writeLine 對 inner pi 寫入時撞 Error: write EPIPE（inner pi 已死），Node unhandled error 讓整個 adapter crash，exit 1。

判定：pi-acp 0.0.x 有兩個上游品質問題——(a) 舊版 handshake 壞掉、(b) inner pi 死亡時不做錯誤處理直接崩潰。用 bunx pi-acp 不 pin 版本，等於每次連線都在賭當下 latest 的品質（19:16 的 crash 就是升版當下引入）。

階段③全靜默 60 秒 → initialize timeout kill（現行狀態，log 不足以定位）

19:17 之後所有嘗試（含 07-07 00:36 最後一次）：process 存活、零 stdout、零 stderr，60 秒整被 kill（= acpConnection.ts:490 的 initializeTimeoutMs ?? 60_000），exit code null。

反證實驗（2026-07-07 實測）：以 app 完全相同的方式（/bin/zsh -lc 'bunx pi-acp'、非 TTY piped stdio、程序存活中讀取）送 initialize，1 秒內收到正確回應（pi-acp 0.0.31，含 authMethods pi_terminal_login）。CLI 路徑完全健康。

剩餘候選原因（log 無法區分，這正是觀測缺口）：

  • H1：initialize 根本沒寫進 child stdin——前端 ndJsonStream→agent_write 鏈路某處靜默失敗（WritableStream 錯誤被 ACP lib 吞掉，request 永遠 pending）。
  • H2：GUI app 環境中 bunx 對 registry 重新解析 latest 時卡住（bunx 不 pin 版本會定期重查 manifest；卡網路 = 無輸出地掛著，與觀測完全吻合）。
  • H3：GUI 環境變數差異讓 pi-acp 啟動後掛住（無法從 log 驗證，因為 handshake trace 預設關）。

為什麼 log 幫不上忙（logger 對 ACP 的觀測缺口）：

  • acp_spawn 不記錄實際 command、resolved shell、PATH 摘要（agent_process.rs:273-286）——連「那次到底跑了什麼指令」都答不出來。
  • acp_trace_in/out 被 trace_enabled gate（預設 false，agent_process.rs:53）——出事當下永遠沒開，handshake 有沒有寫出去、寫了什麼，零紀錄。
  • acp_kill 不記 reason——60s init timeout、使用者手動停止、app 退出，在 log 裡長一樣（10s 與 60s 的 kill 只能靠對時間差猜）。
  • env_path 結果只 eprintln 不進 log（階段①的直接盲點）。
  • acp_stderr 一律 level "debug"——即使內容是 Node crash stack，在 Logs pane 篩 error/warn 完全看不到。
