ACP 統一架構評估

問題：能否把 ACP 做成統一架構，預設提供 claude · codex · pi 三選？· 2026-07-07 · 錨定原始碼 file:line · 姊妹文件：acp-功能分析-2026-07-07.html（缺陷／功能分析）

TL;DR — 可行，而且比想像近

結論：可行；現有程式碼已經替統一架構鋪好約 70% 的路。ACP 本質就是「一個 client 對多種 agent」的抽象，而 yuzora 目前的三層剛好都站在對的邊界上：傳輸層（Rust agent_process）完全 agent-agnostic（只 spawn 任意 command + ndjson 管線，零 pi 邏輯）、協定層用的是標準 Zed ACP（ClientSideConnection + initialize/newSession/prompt/cancel）、啟動 command 已是可注入的，且 Settings 早已存在 preset 欄位（"pi" | "custom"）與 resolveAgentCommand()。

換句話說：今天使用者其實已能用「custom」欄位手打 claude-code-acp 的 command 跑起來。要做的是把它從「一個文字框」升級成「claude / codex / pi 一級選單」。

兩個層級（可分批）：Tier 1 全域 agent 選擇器（S–M、低風險：擴 preset 登錄表＋下拉＋pi 專屬欄位降級）→ 立刻可用的三選；Tier 2 per-session agent（L：連線改為 per-(agent,cwd)、SessionState 帶 agentId）→ 才能「這個 session 用 claude、那個用 pi」，而它與缺陷分析裡的 P10 同一個根因（singleton 連線）。

更新（2026-07-07 查證 npm）：claude 與 codex 都有官方 org 已發布的 adapter。整個 ACP 生態已從 @zed-industries/* scope 遷移到 @agentclientprotocol/*：claude → @agentclientprotocol/claude-agent-acp@0.57.0、codex → @agentclientprotocol/codex-acp@1.1.0，兩者同一組 Zed 核心維護者（cirwin 等）、掛在官方 github.com/agentclientprotocol org。codex 不再是未知數。

⚠ 連帶維護債：專案自己的協定套件也被 deprecated。package.json:56 的 @zed-industries/agent-client-protocol@^0.4.5 已改名為 @agentclientprotocol/sdk（現版 1.2.0）。這是 0.4 → 1.2 的 major 跳版，不是無痛替換——遷移前要查 ClientSideConnection 等 API 是否有 breaking change，並依 major 升級慣例檢查 lockfile 雙版本。此項與統一架構獨立，但建議一併排進去。

兩個仍成立的警訊：① 各 agent 的 auth UX 不同、須實機驗（ACP 的 authMethods 機制已備好，但 pi=env-var／claude=Anthropic key・oauth／codex=OpenAI 登入，流程要真跑）② pi 專屬的 _meta.piAcp 功能（running 點、P9 startupInfo）對 claude/codex 會降級成 no-op，需補 agent-agnostic 後備。

為什麼「已經 70% 到位」

把一次 prompt 的路徑攤開，會發現 pi 的耦合只集中在極少數點，主幹是 agent 中立的。下圖：綠色邊界＝已 agent-agnostic；橘色＝pi 專屬耦合點；紅色＝擋住 per-session 的 singleton。

四個「已經到位」的事實（可查證）

① 傳輸層完全 agent-agnostic。agent_process.rs 的 spawn(command) 只是「用 shell 跑一條字串、把 stdout 逐行 emit、stdin 寫回」，內部無任何 pi 假設。換 agent＝換 command。src-tauri/src/agent_process.rs:75（spawn(command)）· :439（shell_command）· :604（agent_spawn 收任意 command: String）
② 協定層是標準 Zed ACP，不是 pi 私有協定。用 @zed-industries/agent-client-protocol 的 ClientSideConnection，握手是通用的 initialize({protocolVersion:1, clientCapabilities}) → newSession / prompt / cancel / loadSession。claude-code-acp、gemini-cli、codex-acp 講的是同一套協定。src/agent/acpConnection.ts:544（ClientSideConnection）· :594-600（標準 initialize）· :611-643（session/prompt/cancel）
③ 啟動 command 已是可注入依賴。deps.command 可為字串或 thunk，預設才 fallback "bunx pi-acp@0.0.31"。src/agent/acpConnection.ts:586（typeof deps.command === "function" ? …）· :588（fallback）· AgentBridge.tsx:45（command: () => resolveAgentCommand(loadAgentSettings())）
④ Settings 早就有 preset 概念與解析函式。AgentSettings { preset: "pi" | "custom", command, traceEnabled }，resolveAgentCommand() 已在做「preset → command」的分派。今天選 custom 手打 command 就能換 agent；缺的只是把它做成一級選單。src/app/workbench/settingsStorage.ts:8（AgentPreset）· :20-24（AgentSettings）· :78-81（resolveAgentCommand）

已具備 vs 待補缺口

把「統一架構＋三選」拆成 6 個工項。前 5 項小，第 4 項是唯一的架構級改動（且與 P10 同根因）。

G1preset 登錄表：pi | custom → pi | claude | codex | custom半具備S

現況

AgentPreset = "pi" | "custom"；resolveAgentCommand 只有「pi 回死值、否則回 custom 欄位」兩條路。

要做

把 preset 擴成 union，並把「preset → 預設 command」抽成一張登錄表（{ id, label, command, docs }）。resolveAgentCommand 改成查表；SettingsDialog 的 agent 段落把單一 command 文字框換成下拉（選 preset 時帶出預設 command，custom 才可編輯）。零協定改動、零 Rust 改動。

證據 / 落點

  • src/app/workbench/settingsStorage.ts:6（DEFAULT_AGENT_COMMAND）· :8 · :78-81（改查表）
  • src/app/workbench/SettingsDialog.tsx（agent 段落改下拉；目前只 import resolveAgentCommand/loadAgentSettings/AgentSettings）
pin 版本的教訓仍要守：登錄表裡的 command 都要 pin 版本（bunx 不 pin 會 re-resolve latest → 網路 hang／版本漂移，2026-07-06 EPIPE 即此類）。
G2pi 專屬 _meta.piAcp 讀取 → 對其他 agent 降級後備缺口S

現況

sessionInfoFromUpdate 讀 _meta.piAcp.{queueDepth, running} 來驅動「執行中」狀態；P9 規劃讀 _meta.piAcp.startupInfo。claude/codex 不會送這欄。

要做

兩個都是降級即可、非破壞：running 點改為「有 _meta.piAcp.running 就用，否則以 store 既有的 in-flight/streaming 狀態推斷」——否則 claude/codex 跑起來時「執行中」橘點不會亮。startupInfo（P9）對非 pi agent 就是拿不到 → banner 不顯示，天然無害。

證據 / 落點

  • src/agent/acpConnection.ts:700-703（_meta.piAcp.{queueDepth,running}）· :611-620（newSession 讀 _meta 的落點，見 P9）
  • src/state/agentStore.ts:327-342（onSessionInfo 驅動 running）· beginTurn/streaming 已是 agent 中立訊號
G3各 agent auth 流程：機制已備，須逐 agent 實測機制已備M

現況

ACP 的 auth 已標準化：initialize 回 authMethods，程式已 normalizeAuthMethods 並有 trackAuthRequired ＋ 終端登入流程（terminalLoginShellCommand 會用 envPrefix + agentCommand + method.args 組登入指令）。

要做

機制是通用的，但每個 agent 的 authMethods 內容與登入體感不同：pi 走 env-var 憑證（GEOSENSE_API_KEY / YUUZU_API_KEY）、claude-code-acp 走 Anthropic API key 或 Claude 訂閱 oauth、codex 走 OpenAI/ChatGPT 登入。這些無法在 jsdom 驗，要真的各跑一次登入→授權→prompt。

證據 / 落點

  • src/agent/acpConnection.ts:601（normalizeAuthMethods）· :613/632（trackAuthRequired）
  • src/state/agentStore.ts:486-491（terminalLoginShellCommand：envPrefix + command + method.args）
pi 憑證是 env-var 制、只在 ~/.zshrc；env_path.rs 現已搬全 env（非只 PATH），claude/codex 的 key 若也走 env 需確認同樣被帶進 spawn 環境。
G4singleton 連線 → connection-per-(agent, cwd)架構阻斷L= P10

現況（這是關鍵）

AgentBridge 用 useMemo(createStoreBackedConnection, []) 建唯一連線；ensureConnection 開頭 if (agent) return——所以 command 只在首次 spawn 時解析一次。後果：

  • 在 Settings 改 preset 不會即時生效，要等 process 被 kill／重啟。
  • 無法「這個 session 用 pi、那個用 claude」同時並存——全 app 綁一個 agent。

這正是缺陷分析 P10 的同一個 singleton 根因（P10 是「綁首個 cwd」，此處是「綁首個 agent」）。

要做

把連線從 singleton 改為 以 (agentId, cwd) 為 key 的連線池；SessionState 帶 agentId（見 G5），ensureConnection 依 key 找／建連線。Tier 1 可跳過此項（全域單 agent、切換重連即可）；只有要 per-session 多 agent 才必須做。

與 P10 綁定：既然修 P10 本來就要動連線 keying（其 Stage C 就是 connection-per-cwd），建議把 G4 與 P10 一起做——同一塊重構，別分兩次動。

證據 / 落點

  • src/workbench/AgentBridge.tsx:12（useMemo(…, []) singleton）· :45（command thunk 只在首 spawn 求值）
  • src/agent/acpConnection.ts:485（if (agent) return）
G5SessionState 帶 agentId（標示／路由／徽章）缺口S

現況

SessionState 有 model / cwd / running / tone … 但沒有 agentId——無法知道某 session 屬於哪個 agent。

要做

加 agentId 欄位（newSession 時戳上目前 preset）。用途：session header／nav 標示「pi · claude · codex」小徽章、per-session 路由 prompt/cancel 到對的連線。Tier 1 也建議先加（純標示，成本低）。

  • src/state/agentStore.ts:30-45（SessionState 形狀）· :135-143（newSession 建 session 處戳 agentId）
G6capabilities-aware UI（讀 initialize 的 agentCapabilities）選配S

現況

initialize 的回傳目前只取 authMethods，其餘 agentCapabilities（支援的 prompt 內容型別：image/audio、embedded context、slash commands…）被忽略。

要做（可延後）

讀 agentCapabilities，讓 UI 依 agent 能力調整（例：agent 不支援貼圖就不顯示貼圖入口）。三選能跑不靠這項；屬健壯性 polish。

  • src/agent/acpConnection.ts:594-601（目前只從 initializeResult 取 authMethods）

三個 agent 對照

ACP 生態的三個候選，全部有官方 org 已發布 adapter（2026-07-07 npm 查證）。三者同一個 @agentclientprotocol/* scope、同組 Zed 核心維護者——這正是「統一架構」在生態層面的背書：一個 client、同一協定、多個對等 adapter。

建議落地路線

🟢 Tier 1 · 全域 agent 選擇器（S–M，低風險，先做）
G1 preset 登錄表（pi/claude/codex/custom＋下拉）＋ G2 pi 專屬欄位降級（running 後備）＋ G5 SessionState.agentId（純標示）。
產出：使用者可在 Settings 選 claude / codex / pi，全 app 使用選定 agent（切換需重連一次）。不動 singleton、不動協定、不動 Rust。登錄表用新名（@agentclientprotocol/claude-agent-acp@0.57.0、@agentclientprotocol/codex-acp@1.1.0、pi-acp@0.0.31）；claude/codex 各先跑通一次 auth。
🔴 交會點 · 與 P10 一起做（L）
G4 connection-per-(agent, cwd)。這塊重構同時解掉缺陷分析的 P10（切 workspace 清空）——兩者都是 singleton 連線的病。既然 P10 遲早要動連線 keying，把「per-cwd」與「per-agent」一次做完最省。此步完成後，Tier 2 才有地基。
🟠 Tier 2 · per-session agent（L，依賴上一步）
每個 session 各自選 agent、並存運行（pi 一個 tab、claude 另一個 tab）。UI：new-session 選 agent、nav 依 agentId 分組／徽章（可與缺陷分析 F2「workspace agent 徽章」合流）。
🟡 Polish（選配）
G6 capabilities-aware UI（依 agentCapabilities 調 UI）＋ G3 各 agent auth 實機驗收（gui-acceptance，逐 agent 跑登入流程）。
🔴 獨立維護債 · 協定套件遷移（可任意時點，別漏）
package.json:56 的 @zed-industries/agent-client-protocol@^0.4.5 已 deprecated → 遷移到 @agentclientprotocol/sdk@1.2.0。0.4 → 1.2 major 跳版，但已實測 API 表面相容（2026-07-07 拆包 diff）：程式碼觸及的每一處皆一字不差——建構子 (toClient, stream)、initialize / newSession / loadSession / prompt(PromptRequest) / cancel（新檔仍在 ClientSideConnection，prompt 於 acp.d.ts:1184）、ndJsonStream 簽名；import 5 符號（ClientSideConnection, ndJsonStream, AnyMessage, Client, Stream）strict tsc exit 0；Client 介面回傳 Promise→MaybePromise 屬相容放寬。實質 drop-in。唯一殘留：schema.* 型別欄位級漂移，換 dep 後跑 bun run build＋bun run test 蓋章即可（協定仍 protocolVersion:1，機率低）。依 major 慣例順手查 lockfile 雙版本。與統一架構技術上獨立，建議同波處理。

決策：你要哪一種「統一」？

若目標是「有得選、切換方便」 → 只做 Tier 1 即可，成本小、風險低，且不碰 P10 的高風險重構。多數情境這就夠。
若目標是「同時跑多個 agent、各 session 各自為政」 → 必做 Tier 2，而它綁 G4＝P10 重構。建議把 P10 修復與此當成同一個 milestone 規劃，別各修一半。
