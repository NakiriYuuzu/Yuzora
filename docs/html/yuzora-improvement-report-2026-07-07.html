現況一句話

yuzora 是一個健康度偏高的 Tauri 2 + React 19 桌面 IDE：里程碑文件齊全、前端測試密度高、Rust 依賴管理有紀律（連 tokio features 都附取捨註解）。它的弱點不在「寫出來的程式碼」，而在「沒有任何自動化機制保證程式碼持續是這個品質」。

做得好的（不需要動）
  • 程式碼衛生：全 src/ 掃描 TODO/FIXME/HACK = 0、console.log = 0、any 僅 1 處。
  • 測試文化：前端測試 18,058 行 vs 源碼 23,172 行；Rust 端 25/28 檔有 #[cfg(test)]。
  • tsconfig 已開 strict + noUnusedLocals + noUnusedParameters。
  • docs/superpowers 按里程碑組織、命名一致；Cargo.toml 依賴附取捨理由註解。
三大結構性風險
  • 流程風險（高）：無 CI、無 lint、無 pre-commit——品質完全靠「人工／agent 跑 bun run build／test／cargo check」的紀律，紀律一鬆就劣化，且無法回溯是哪次變更弄壞的。
  • 執行期風險（高）：Rust 端 panic! 集中在 db_service.rs(6)、git_service.rs/git_log.rs/fs_service.rs(各3)；若發生在 watcher/pty/agent 背景執行緒，會直接拖垮整個 process 而不是回傳 Err 給前端。
  • 變更管理風險（中）：101 檔／9,195 行插入的未提交批次（含 i18n 重構、SSH/DB 新 service、9 份舊結案報告刪除），任何新工作都疊在這座未定案的山上。

Rust service 的 unwrap()/expect() 熱點分布

一句話：風險集中在 git 與 agent/lsp 三條主要 IPC 路徑上，優先審查報酬最高。

4加 ESLint（flat config）高小至少 typescript-eslint + eslint-plugin-react-hooks。tsc strict 抓不到 hooks 依賴陣列、exhaustive-deps 這類問題。

詳細分析

① 工具鏈與 CI（嚴重度：高）
  • 無 .github/、無 ESLint／Prettier／Biome 設定、無 clippy.toml／rustfmt.toml、無 pre-commit hook。僅有 .editorconfig。
  • package.json scripts 只有 dev/build/test 系列，沒有 lint、format、獨立 typecheck。
  • Rust 測試存在（25/28 檔有 #[cfg(test)]）但不在任何自動化流程裡——CLAUDE.md 的驗證指令只有 cargo check。
  • 沒有 cargo audit／依賴健康掃描排程。

建議最小 CI：macos-latest 一個 job：bun install → bun run build → bun run test → cargo clippy -- -D warnings → cargo test。Rust 編譯可加 Swatinem/rust-cache 控制時間。

注意：統計含測試模組內的 unwrap（測試裡 unwrap 是合理的），所以真實風險數低於帳面值。但 panic! 與背景執行緒內的 unwrap 需要逐點確認——Tauri command handler 裡的 panic 會被框架接住回傳錯誤，helper thread 裡的 panic 則直接讓執行緒死掉甚至拖垮 process。

建議做法：先加 [lints.clippy] unwrap_used = "warn"（allow in tests），跑一次 clippy 產出清單，按「是否在背景執行緒」分類，再決定改 ?／anyhow／明確錯誤回傳。

③ 測試缺口（嚴重度：中）
  • src/app/panels/PreviewPanel.tsx：676 行、零測試，且直接 import @tauri-apps/api/core。
  • src/features/logs/logQuery.ts、userAction.ts：M6 新功能核心邏輯，零測試。
  • src/lib/workspaceActions.ts、src/lib/saveSuppress.ts：跨面板共用邏輯，無直接測試。
  • AgentZonePanel.tsx（1,255 行）對應測試僅 464 行，覆蓋深度偏低。
  • jsdom 測不到的部分（watcher/tab/dialog）已有 gui-acceptance skill 制度補位，這點是好的。
④ 前端架構（嚴重度：中）
  • IPC 雙軌：src/lib/ipc.ts 是官方封裝層，但 AppShell.tsx、PreviewPanel.tsx、features/logs/*、agent/acpConnection.ts、lib/platform.ts 直接呼叫 @tauri-apps/api/core。統一後 mock 策略與錯誤處理才會一致。
  • 17 個 zustand store：無集中索引文件；agentStore.ts 直接 import terminalStore、uiStore 形成跨 store 耦合。建議至少補一份 store 依賴關係說明，或收斂 orchestration 到 bridge 層（專案已有 AgentBridge／SshBridge 的模式可循）。
  • 巨型元件：SettingsDialog.tsx 1,725 行橫跨 LSP/Agent/Logs/Editor 四個分頁——四個測試檔已經按分頁切了，元件本身也應按分頁拆檔。
⑤ 文件與變更管理（嚴重度：中）
  • docs/html/ 的 9 份舊結案報告目前是「已刪除未提交」狀態，同時新增了 3 份 07-07 的新報告。若刪除是預期的清理，建議儘快 commit 定案；若不是，需要救回。
  • 工作區 101 檔／9,195 行插入的未提交批次是所有後續工作的地基，建議依主題切成數個 commit（i18n 重構／SSH+DB service／editor 主題／報告清理…）。
  • .superpowers/sdd/ 約 200 個歷史過程檔已標註「僅供考古」，屬刻意保留，非待辦。
⑥ 依賴健康（嚴重度：低）
  • 整體新穎：React 19、Tailwind 4、Vite 8、Vitest 4、TypeScript 6、zustand 5。
  • Rust 側版本釘選明確、關鍵依賴附取捨註解（如 tokio features）——良好實踐。
  • 待查證：lucide-react ^1.23.0 的版本序與該套件慣常 0.x 不同，建議核對 registry；radix-ui／shadcn 合併套件的版本穩定性值得留意。
  • CodeMirror 12+ 個語言套件屬正常組合，但可留意 bundle 是否已做語言 lazy-load。

建議執行路線（三波）

第一波：把手動紀律變成機器保證（約半天～一天）

  1. 確認 docs/html 舊報告刪除意圖 → 將 9,195 行批次切成邏輯 commit（需你明確下令）。 → 驗證：git status 乾淨
  2. 建 .github/workflows/ci.yml（build + vitest + clippy + cargo test）。 → 驗證：push 後 CI 綠燈
  3. 加 ESLint flat config + lint script；加 Cargo [lints]。 → 驗證：bun run lint、cargo clippy 通過或產出已知清單
  4. 加輕量 pre-commit hook。 → 驗證：commit 時自動跑 tsc

第二波：執行期穩定性（約一～兩天）

  1. 以 clippy unwrap_used 清單為基礎，專項審查 db/git/fs service 的 panic! 與背景執行緒 unwrap。 → 驗證：cargo test + gui-acceptance 復驗關鍵流程
  2. PreviewPanel、logQuery/userAction 補測試。 → 驗證：新測試紅→綠

第三波：結構性重構（可分批、低優先）

  1. IPC 呼叫路徑統一收斂進 src/lib/ipc.ts。
  2. SettingsDialog 按分頁拆檔；lsp_service/lsp_download 拆子模組。
  3. 補 store 依賴關係文件（或畫進 CLAUDE.md 重要路徑表）。

每一項都刻意配了驗證方式，方便沿用專案既有的「plan → implement → verify」節奏。第三波不急——有了第一波的 CI 護欄之後再做重構，風險才是受控的。
