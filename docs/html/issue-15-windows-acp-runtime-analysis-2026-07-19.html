二、現在的失敗鏈

失敗發生於 process spawn 前，因此 provider authentication、ACP initialize 與 session creation 都尚未開始。

三、根因與次要因素

Primary cause

Curated command model 把「Agent 身分」與「啟動 runtime」綁死

src/lib/agentPresets.ts 的 Verified／Latest command 都直接保存完整 bunx package@version 字串，沒有 package spec、bin name 或 runtime-specific launch templates。

Boundary

Backend preflight 是 verifier，不是 resolver

agent_process.rs 只接收 command/cwd，能可靠判斷第一個 token 是否存在，但不知道這是 curated Pi、Claude、Codex，無從選擇等價 Node 或 Deno invocation。

Identity risk

不能在 Rust 端悄悄替換字串

Frontend 以 command + cwd 作 connection key，並把 command route 用於 auth、prewarm、restore 與 custom fingerprint。若 backend 暗換命令，邏輯身分與實跑 process 會漂移。

Secondary

Authentication 可能是下一關，但不是目前第一個根因

它無法解釋現在的 bunx not found，因為 adapter 尚未 spawn；但 resolver 修好後，auth-required provider 仍會走 terminal login。

Windows blocker

Terminal login command builder 是 POSIX-only

agentStore.ts 固定組出 -c、env NAME=… 與單引號 quoting；Windows 預設 cmd.exe 需要不同參數、env 與 quoting contract。共用 resolved command 只是必要條件，不是完整修復。

八、建議責任切分

核心原則：把「使用者選的是哪個 Agent／版本模式」與「這次 connection attempt 實際用哪個 runtime 啟動」拆成兩個明確欄位；trusted curated launcher 必須使用 App PATH 解析出的絕對路徑，避免 workspace-local 同名檔案劫持 background prewarm。

新增安全邊界：既有 Windows preflight 會搜尋 workspace cwd，符合 shell 執行使用者 command 的語意；但自動執行的 curated preset 不得從 cwd 選 bunx.cmd／deno.exe／npm.cmd。resolver 必須排除 cwd、檢查 runtime 版本，並把絕對 launcher path 帶到實際 spawn。

Frontend descriptor：adapter 單一真相

  • Metadata 先行：AgentDescriptor 必須新增 package name、bin name、Verified exact contract、Latest compatibility policy 與 runtime-specific invocation；不可再從 bunx … 字串反向解析，也不可另建隱性重複表。
  • 最佳插入點：agentRouter.routeFor() 已是 async，可在 ensureSub() 前把 trusted logical route 轉成 resolved launch route。
  • 安全搜尋：trusted launcher 只從 App PATH/PATHEXT 解析並使用絕對路徑；workspace cwd 只保留給使用者明確輸入的 custom command。
  • 可用候選順位：Bun → Deno → Node 只在「launcher 完整且版本相容」的 candidates 間排序。高順位缺 component 或過舊時必須繼續嘗試下一順位；只有全部 rejected 才回聚合 prerequisite。
  • Node invocation 定案：建議只採 npm exec --yes --package=<spec> -- <bin>，因此 Node candidate 要求 node.exe + npm.cmd；不在同一版本同時支援 npx 與 npm exec，避免 interface/test 分叉。
  • Latest 不是 Verified：@latest 的 adapter version 與 engine 會移動，不能沿用 Verified 的 min-runtime。successor plan 必須選擇啟動前查 package metadata，或明確標示 best-effort 並把 engine mismatch 分類為 package compatibility error。
  • 同一真相：prewarm、newSession、該次 loadSession connection、router cache key 與 process spawn 必須拿同一 resolution。這不新增「跨重啟保存 latest 精確版本」承諾；Session Index 仍依現有 ADR 只存 agentId／custom fingerprint。
  • Windows auth/login：terminal login 需另有 platform-aware launch contract；現有 POSIX -c／env／quote builder 不可只替換 command。至少要驗證預設 cmd.exe 與支援的 custom shell。
  • custom 安全：trustedAgentId === null 時完全跳過 resolver；既有 raw-command SHA-256 fingerprint 不變。
  • 缺少或過舊：回 stable typed prerequisite error，列出各 runtime 的 rejected reason、實際版本與最低需求，而不是 generic command-not-found。
  • 冷啟動 timeout：npm exec 首次下載不能與既有 15 秒 ACP initialize timeout 混成同一階段；需有 cold-start budget、stage diagnostics 與慢 proxy／離線測試。
