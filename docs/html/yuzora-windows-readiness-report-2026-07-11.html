yuzora Windows 就緒度報告

2026-07-11 · 分析基準：工作樹磁碟現況（含在途未提交工作）· 出貨承諾：macOS Apple Silicon ＋ Windows x64（MVP spec 測試策略章 L363）· 姊妹篇：yuzora-architecture-design-report-2026-07-11.html

TL;DR

Windows x64 是 spec 明文的出貨平台，但九天的開發全程沒有 Windows 實機 — M1 到 M5 的驗收一貫標註「Windows 側 deferred」（M5 design L232 明文承認，與 M1–M4 慣例一致），所有 cfg(windows) 分支從未被任何 Windows 編譯器碰過（CI 的 rust job 固定跑 macOS）。

盤點結果：5 個 blocker、5 個 degraded、2 個 cosmetic。好消息是程式碼層的平台分支寫得誠實 — spawn 一律經 cmd.exe /C（.cmd shim 可解析）、快捷鍵全部 meta/ctrl 雙相容、視窗 chrome 刻意讓 Windows 走原生標題列、lsp 安裝計畫有 .exe/.cmd/Scripts 分支。壞消息是三個核心承諾未兌現：askpass 在 Windows 是空 stub（git push/pull 無認證 UI）、Job Objects 未實作（只有 taskkill /T 快照式清樹）、spec 承諾的 L3 Windows E2E blocking gate 整層不存在。

工作量判斷：完整兌現 spec（L3 harness＋Job Objects＋askpass＋簽章）約 L 級（2–3 個工作週）；「可跑、降級可接受」的最低出貨門檻約 M 級（一週內）＋至少一台 Windows 實機。第一步是小時級的：CI 加 windows-latest cargo check — 立刻把「從未編譯過」變成「至少編譯過」。

為什麼會走到這裡：結構性成因

這不是疏忽累積，而是一個明知的、有書面記錄的取捨反覆執行九天的結果：

  • M4 計畫（docs/superpowers/plans/2026-07-05-yuzora-m4-terminal-preview.md:166）原文：「Job Object 具體 API 走 windows crate 哪個 feature——無 Windows 實機，標註待驗」— [SPIKE 驗證] 標記從計畫一路帶進程式碼（process_kill.rs:32-35）。
  • M5 design（2026-07-06-yuzora-m5-agentzone-design.md:232）：「Windows 側 deferred（開發環境無 Windows，與 M1–M4 慣例一致）」；其風險表 L246 把補救綁在「沿用 MVP spec L3 規劃」— 而 L3 至今整層未建。
  • CI（.github/workflows/ci.yml:33-52）rust job 固定 macos-latest，release（release.yml:29-44）雖有 windows-latest 矩陣，但只在打 tag 時第一次真正編譯 Windows — 編譯錯誤的發現點被推遲到發版當下。

結論：債務是自覺借的、單據齊全 — 現在的問題只是「何時還、還多少」。這份報告的差距盤點就是還款清單。

已就緒面 — 程式碼審閱層面正確的部分

公平起見，下列設計顯示團隊在「無實機」的約束下仍持續為 Windows 鋪路，這些是出貨時不需要重做的：

  • spawn 一律經 shell 轉發：agent 與 dev server 的 shell_command 在 Windows 走 cmd.exe /C（agent_process.rs:439-453、process_service.rs:492-507）— .cmd shim（bunx/npm）與 PATHEXT 由 cmd.exe 解析，避開了「CreateProcess 不解析 .cmd」的經典坑。
  • 預設 shell 解析鏈平台正確：ComSpec → fallback cmd.exe（pty_service.rs:580-586）；login 旗標 -l 只在 unix 加。
  • 快捷鍵全面雙相容：palette ⌘K/Ctrl+K、composer 送出、terminal Ctrl+` 全部 metaKey || ctrlKey 或正確的 ctrl-only 寫法；唯一需要平台判斷的 git 多選 modifier 也有 navigator.platform 分支（gitChangeSelection.ts:162-165）。
  • 視窗 chrome 是刻意決策：titleBarStyle Overlay／trafficLightPosition 是 macOS-only 概念，Windows 自然落回原生標題列；前端 showsNativeTrafficLights() 守衛了 drag 帶與 20px 位移（platform.ts:12-14、AppShell.tsx:228-239）。
  • LSP 安裝計畫的平台分支大致齊全：.exe 後綴、npm .cmd shim、venv Scripts 目錄都有 windows: bool 分支（plan.rs:166-168,225-228,247-249）— 只缺 zip 解壓（B3）。
  • SSH／known_hosts 平台中立：dirs crate 取 home、純 JSON 檔、無 unix socket 或 chmod 假設（ssh_service.rs:74-78）。
  • 設定不寫死 unix shell：shellPath/shellArgs 預設空字串，fallback 下沉到 resolve_shell（settingsStorage.ts:14-31）。
  • 檔案原子寫入有 Windows 意識：db_profiles.rs:246 註解自陳 Windows 無可攜目錄 fsync，改靠 NamedTempFile::persist 的平台 replace 語意。

建議工作計畫（四階段）

Phase 0 — 立即（小時級）
把「從未編譯」變成「至少編譯」ci.yml rust job 改 matrix（macos＋windows＋ubuntu 的 cargo check --all-targets；windows 可先只 check 不 test 以控時長）。這一步零風險、立即攔截所有 cfg(windows) 編譯錯誤 — 在任何人碰 Windows 程式碼之前先做。
Phase 1 — Blocker 修復（M 級，需一台 Windows 實機）
B3 zip 解壓（最便宜，先做）→ B5 路徑前綴（dunce＋basename helper，可 TDD）→ B1 askpass named pipe → B2 Job Objects（[SPIKE 驗證] 的原始待驗項，最後做因為要實機調試 handle 生命週期）。每項都先寫 Windows 上可跑的測試。
Phase 2 — Degraded 收斂（S 級 ×4）
D1 preflight（where.exe）、D4 ConPTY/WebView2 實機 smoke、D5 MOTW 驗證、C1/C2 文案與截圖。D2 env 機制與 D3 簽章視產品決策（憑證採購）另排。
Phase 3 — 驗收制度（與出貨門檻掛鉤）
最低門檻：把 gui-acceptance checklist 在 Windows 實機跑一輪（等同 macOS 的 L4 慣例）＋ release smoke（L5：安裝檔可裝可開）。完整門檻：補 spec 承諾的 L3 tauri-driver E2E（L 級，可獨立排程，不阻擋首個 Windows beta）。

關鍵排序邏輯：驗證基建（Phase 0）先於修復（Phase 1），因為沒有編譯 gate 的修復無法防退化；B2 Job Objects 放 Phase 1 最後，因為它是唯一必須「實機在手」才能開工的項目。
