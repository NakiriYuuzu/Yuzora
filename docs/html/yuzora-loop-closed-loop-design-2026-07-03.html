Design Doc

Yuzora 任務閉環（yuzora-loop）

建立於 2026-07-03 · Fable-5 bootstrap session · 供 Opus / Sonnet / Haiku 後續維護使用 · v1.1（同日依使用者指示修訂）

TL;DR

本專案自今日起，所有任務必須依 yuzora-loop 閉環執行：TRIAGE → 提問 → 規劃 → 執行 → 對抗 review → 驗收 → response。閉環把原本依賴大模型判斷力的部分外置成固定閘門、run file 模板與證據要求，讓較小模型也能維持同等嚴謹度。強制機制共三層（CLAUDE.md 常駐宣告、skill 自動觸發、UserPromptSubmit hook 每訊息提醒），並以 LEARNINGS 棘輪（可自主加嚴、放寬需人批准）實現安全的自我改進。v1.1 修訂：撰寫與對抗 review 交由 opus subagent 執行（controller／worker 分離）、git 開放唯讀、worktree 可詢問啟用（預設不用）。

為什麼需要閉環

今天之後，本專案的維護只能使用 Opus / Sonnet / Haiku。過去 M1、M2 的品質來自「大模型＋subagent 對抗 review」的流程。v1.0 初版在完全無 subagent、無 git 的限制下設計；v1.1（同日使用者批准）恢復受控派工：subagent 僅限 opus、git 僅限唯讀、worktree 預設不用。不變的設計哲學是：把判斷力外置成結構——controller 照模板派工、照閘門過關、照指令貼證據，可靠度就不再依賴單一模型的大小。

閉環流程與回饋路徑

一句話：前進要過閘門，出錯必回 P3，經驗只進不出（棘輪）——這三個約束構成閉環。

設計原則

判斷力外置弱模型的弱點在裁量，不在執行。所有裁量點都改寫成二元檢查：有沒有貼輸出、有沒有 ≥3 條反證、E 清單有沒有在寫碼前固定。
證據先於宣稱沒貼實際指令輸出就不得寫 PASS。這是從 superpowers verification-before-completion 沿用的原則，也是 blog 所說「verification 要可量化」的落地。
run file 即記憶git 僅唯讀：變更檔案清單＋P2 baseline＋git diff 併用構成對照組；漏記一個檔案＝review 出現盲區，所以「動一個記一個」仍是硬規則。
棘輪式自我改進參考 recursive self-improvement 一文：自我改進需要人類監督。故 LEARNINGS 只允許模型自主「加嚴」，任何「放寬」都必須由使用者在對話中批准。

五條鐵律（v1.1）

#鐵律執行方式
1撰寫與對抗 review 交給 subagent主對話為 controller（規劃、派工、驗證、裁決）；親手只寫記帳文件（run file／LEDGER／LEARNINGS／brief），其餘撰寫一律派 opus subagent
2Subagent 僅限 opus派工必帶 model: opus（禁 sonnet／haiku）；撰寫者與 reviewer 必須是不同 subagent
3Git 僅限唯讀status／diff／log／show／blame 可用（baseline 與 review 善用 git diff）；寫入類一律禁止——唯使用者於對話中明確下令才執行（使用者覆寫，非閉環授權）
4Worktree 預設不使用任何任務可在 P1 詢問是否啟用（大範圍／高風險應問，列 BLOCKING）；未詢問或未獲同意＝不使用
5證據先於宣稱沒貼實際輸出不得寫 PASS；subagent 的證據 controller 仍須親自重跑驗證

階段核心動作閘門條件
TRIAGE判斷要不要進閉環；純問答標 LOOP-EXEMPT 直接回答拿不準一律進閉環
P1 提問需求改寫成可驗證 R1..Rn；先查 LEDGER／LEARNINGS／specs／sdd 再問；不確定點分 BLOCKING（問使用者並等待）與假設（選預設值續行）G1 BLOCKING 清空
P2 規劃先跑 baseline battery；步驟表每步自帶 verify；驗收證據清單 E1..En 於寫碼前固定；>7 步就拆 runG2 每步有 verify＋E 清單固定＋baseline 已貼
P3 執行Controller 寫 brief 派 opus subagent 撰寫（一次一派、紅燈不疊加）；回報後 controller 親自復驗每步 verify；動一檔記一檔；計畫外改動回 P2 補計畫修正；行為變更先寫失敗測試（連續閘門內嵌於每步 verify）
P4 對抗 review未參與撰寫的獨立 opus reviewer subagent（不給作者自辯）；用 git diff＋磁碟重讀；對每條 R 做具體反證；固定 rubric（邊界／時序／state／主題 i18n／scope／孤兒）；controller 逐條裁決 findingsG4 dry pass＋反證 ≥3＋裁決紀錄
P5 驗收按變更面跑 battery 並貼輸出尾段；與 baseline 比對，任何新紅燈都回 P3；E1..En 逐項銷帳G5 全部 E 有證據且 PASS
P6 收尾run file 收尾＋LEDGER 一行＋自我改進三問＋繁中回覆（結論先行）LEARNINGS 檢視完成

run file 的角色

每個任務複製 .claude/skills/yuzora-loop/template.md 到 .claude/loop/runs/<日期>-<slug>.md。模板的段落順序＝閘門順序，前段沒完成不得動後段。它同時是：計畫書（P2）、變更追蹤（P3，與唯讀 git diff 互補）、派工與裁決紀錄（P3／P4）、驗收證據（P5）、交接文件（P6）。首個示範樣本：runs/2026-07-03-yuzora-loop-bootstrap.md（即閉環自身的建立，dogfood）。

三層強制機制

層機制作用時機特性
1CLAUDE.md（專案根目錄）每個 session 常駐 context宣告「最高優先」；即使 skill 沒觸發也看得到鐵律與路徑表
2yuzora-loop skill description模型判斷任務相關時自動觸發description 刻意寫得「pushy」，涵蓋修 bug／加功能／改樣式等所有說法
3UserPromptSubmit hook每一則使用者訊息deterministic：不依賴模型判斷，由 harness 注入提醒文字

三層互為備援：haiku 級模型可能漏看其中一層，但三層同時失效的機率低。注意 hook 需要重啟 session（或開 /hooks 重載）後才生效——settings watcher 只監看啟動時已有設定檔的目錄。

與參考文章的對應

Getting started with loops（claude.com blog）
  • Turn-based loop＋明確 success criteria → P2 的「E 清單寫碼前固定」與每步 verify，對應「specificity helps Claude reach solutions faster」。
  • Verification 編成 skill、可量化 → P5 battery 表＋gui-acceptance 實機驗收（起 app、互動、留截圖），不允許「宣稱完成」。
  • Secondary reviewer 較不偏頗 → v1.1 起直接實現：P4 由未參與撰寫的獨立 opus reviewer subagent 執行（fresh context、不給作者自辯），controller 逐條裁決並留紀錄。
  • Match primitives to tasks、別過度工程 → TRIAGE 豁免純問答；>7 步強制拆 run。
Recursive self-improvement（Anthropic Institute）
  • 瓶頸在 review 而非產碼 → 閉環把最重的結構（反證、dry pass、rubric）都壓在 P4/P5，而不是 P3。
  • 自我改進需要人類監督 → P6 三問＋LEARNINGS 棘輪：模型可自主「加嚴」與記錄陷阱；「放寬」只能提案，由使用者批准。這是把文中的 oversight 原則縮小到專案尺度。
  • Executing well-specified experiments 是強項 → 所以閉環的工作就是把每個任務變成 well-specified：R 清單、E 清單、baseline、battery。

通過的反證（節選）
  • 強制性：三層機制互為備援，單層失效不致漏套閉環。
  • 無 git（v1.0 當時）：協定全文與 battery 掃描無任何 git 依賴；v1.1 起改為唯讀允許。
  • 敷衍風險：G4 要求 ≥3 條具體反證＋rubric 勾選＋dry pass，「looks good」無法過閘。

維護指南

  • 調整規則：加嚴——直接在 LEARNINGS 加 [tighten] 條目即可；放寬——只有使用者在對話中明確批准後，才能修改 SKILL.md 或在 LEARNINGS 條目標註放寬。
  • 嫌 hook 吵：刪除 .claude/settings.json 的 UserPromptSubmit 段即可，仍保有 CLAUDE.md＋skill 兩層強制。
  • run 檔累積：LEDGER 是索引，舊 run 檔可歸檔搬移，但 LEDGER 行保留。
  • 首次實測：建議下次以 Opus 或 Sonnet 跑一個小任務，觀察閘門遵循度，結果回寫 LEARNINGS。
