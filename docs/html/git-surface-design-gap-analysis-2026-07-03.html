Design Gap Analysis

Git Surface：設計 vs M2 實作交叉驗證

Yuzora · 2026-07-03 · 對照 docs/design/project/Yuuzu Workbench.dc.html（Claude Design handoff bundle）

TL;DR

M2 交付的是 Git 的「功能核」，但呈現層只實作了設計稿六個 Git 表面中的兩個。設計稿把 Git mode 定義為一個 JetBrains 式的完整 surface：主區域預設落在 Log（commit graph＋commit 列表＋commit details 三件套），輔以 Local changes、Console 兩個 tab，header 右側有 branch pill 與 Fetch/Pull/Push；sidebar 是一張完整的 commit 工作卡（訊息輸入＋Commit/Review diff/hold-to-discard）加上 staged/changed 逐檔清單；另外還有一個獨立的 Diff viewer modal。

實作現況：Log 與 Console 是 disabled 空殼，主區只有 Local changes 一個可用視圖；sidebar 只剩 branch chip＋兩個計數；Diff modal 不存在；Fetch/Pull/Push 被塞進 StatusBar 的 BranchPopover 底部。這就是「根本變成不一樣的設計和呈現方式」的來源。好消息是：gitStore、runOp、CodeMirror diff、branch 操作等底層都已完備，缺的幾乎全是呈現層——建議按 Phase A→E 五個階段補齊（詳見 §6），其中 Log tab（Phase C）是決定觀感的核心工程。

1. 設計稿對 Git mode 的完整要求

設計稿的 Git mode 由六個表面組成。關鍵前提是：切到 Git mode 時，editor 主區「整個被 Git surface 取代」（設計稿註解原文：“MAIN REGION … — Git mode swaps to the log”，dc.html:350），而不是在 editor 旁邊多開一塊面板。

設計（左）vs M2 實作現況（右）— Git mode 版面對照

要點：設計的主區預設是 Log 三件套（graph＋列表＋details），實作只有 Local changes；設計的 sidebar 是完整 commit 工作流，實作只剩兩行計數；Fetch/Pull/Push 設計放在主區 header，實作藏進 StatusBar 的 popover。示意圖為結構對照，非 1:1 像素。

3. 逐表面詳細比對

G1  Log tab：設計的主角，實作中完全不存在

設計：Git mode 進場預設就是 Log（gitTab: 'log'，dc.html:1625）。組成：

  • Filter toolbar（40px）：文字搜尋（"Filter by message, author or hash…"）＋ User 下拉（frost glass 選單）＋ Date 下拉。
  • Commit graph：72px 固定寬 SVG，多 lane 彩色分支線（藍/綠/橘）、節點圓圈、merge 曲線；與右側列表逐列對位（32px 行高）。
  • Commit 表格：uppercase 表頭 Commit／Author／Date；每列 refs chips（branch/tag 標籤）＋subject＋author 色點＋相對日期。
  • Commit details（右欄 240px）：hash chip（藍底 mono）＋copy 鈕、serif 15.5px subject、avatar＋作者＋email、committed 日期、parents、Changed files 清單（badge＋add/del）＋總計，底部 Checkout／Compare／Cherry-pick／Reset main to here…（Reset 為 hold-to-confirm 危險樣式）。

實作：GitPanelTabs 裡 Log 只是一顆 disabled 的 TabsTrigger，連 TabsContent 都沒有。Rust 端 git_service.rs 也還沒有 commit log 查詢的 IPC command。

影響：這是「呈現方式不一樣」的最大來源——設計中 Git mode 的第一眼是 commit 歷史圖，實作的第一眼是工作樹檔案清單。

G2  主區 header 缺 branch pill 與 Fetch／Pull／Push

設計（dc.html:734-740）：tab strip 右側依序是 branch pill（8px 藍點＋mono branch 名＋chevron，點擊開 branch popup——與 StatusBar 共用同一個 toggleBranchMenu）與 Fetch／Pull／Push 三顆 30px icon 鈕。這讓日常同步操作停留在 Git surface 內、單擊可達。

實作：tab strip 右側空白。Fetch/Pull/Push 要先點 StatusBar 左下角的 branch chip 打開 popover、再往底部找三顆按鈕——兩層深、且和「切分支」動線混在一起。

補法：把 BranchPopover 的觸發面再包一個主區 header pill（Popover anchor 換位即可），三顆 icon 鈕直接呼叫現有 runOp("fetch"|"pull"|"push")——功能層零新工作。

G3  Sidebar Git nav：設計是完整 commit 工作流，實作只剩兩行計數

設計（dc.html:177-243）：sidebar 在 Git mode 是可以「不離開鍵盤就完成一次 commit」的工作面——

  • 頂部卡片（--yz-panel、12px 圓角）：branch pill、ahead/behind（藍 #2456cc／橘 #c8521f mono）、amber「N changed」pill、分隔線、commit message input（32px）、按鈕列＝Commit {stagedCount}（主鈕）＋Review diff（開 diff modal）＋hold-to-discard（32px 紅框 icon 鈕，長按填滿確認）。
  • STAGED 區：uppercase 標籤＋計數＋「Unstage all」；逐檔列（badge、name＋dir、+add／−del、stage toggle icon）。
  • CHANGED 區：同結構＋「Stage all」。

實作：branch chip＋「Staged N」「Changes N」兩顆摘要鈕（點擊只是跳選主區清單的第一個檔案）。無輸入框、無清單、無按鈕。

連帶效應：因為 sidebar 沒做，commit box 被塞進主區 LocalChangesTab 左欄底部——這又造成 G6 中「主區左欄與設計不符」的偏移。兩個 gap 是同一個根因。

G4  Console tab：git 指令歷史無處可看

設計（dc.html:918-932）：深色 terminal 風（--term-bg、mono 12px）。每個條目：狀態色點＋$ git pull --rebase（綠 --term-lime）＋右對齊時間戳，其下縮排 16px 的輸出行。空狀態文案「No git commands run yet」。原型中所有 git 動作（pull/push/checkout/cherry-pick/commit）都會 gitLog() 寫入。

實作：disabled 空殼。不過 gitStore.runOp 是所有 mutation 的統一入口，補一個 ring buffer（cmd、輸出、tone、時間）＋一個唯讀 UI 即可，是五個缺口中最便宜的。

G5  Diff viewer modal：三個設計入口沒有落點

設計（dc.html:1393-1465）：1040px×88vh 置中 modal，backdrop 模糊。header：diff icon＋serif 標題（「Diff · {來源}」）＋sub（描述 working tree 或 commit hash）＋Unified/Split segmented toggle＋close。左 236px 檔案清單（badge＋name），右 diff 本體。

三個入口（dc.html JS）：

  • reviewDiff（sidebar 按鈕）→ openDiffWork()：整個 working tree 的 diff。
  • openDiffCommit(commit, fileIdx)：Log 的 commit details 點檔案、或「Compare」按鈕。
  • openDiffCurrent()：從 editor 檢視目前檔案的 diff（切到 git mode＋local tab 再開 modal）。

實作：無 modal。working-tree diff 只能在 Local changes 右欄逐檔看；commit diff 根本沒有資料源（依賴 G1 的 Rust 端）。DiffView.tsx（CodeMirror MergeView/unifiedMergeView）本身可直接複用進 modal。

G6  Local changes tab：結構對了，細節有三處偏移
  • Diff header 資訊：設計是 badge＋mono 路徑＋語言標籤＋+N（綠）／−N（紅）統計；實作是 badge＋路徑＋Unified/Split toggle，缺 lang 與統計。
  • Unified/Split toggle 的位置：設計中 local tab 的 diff 是 unified-only，split 切換屬於 Diff modal 的 header；實作把 toggle 放進了 local tab。功能上是加分項，但位置與設計不符（見 §7 決策 D2）。
  • 左欄形態：設計是單一平面清單（因為 staging 的分組視圖在 sidebar）；實作是 Staged/Changes 兩個 section＋底部 commit box。這是 sidebar 缺席（G3）的補償性設計——若 Phase E 落地，需回頭決定左欄收斂方式。

值得肯定：312px 欄寬、--paper-1 底、hover 顯示的 stage/discard icon 鈕、selected 樣式等 token 級細節與設計語言一致。

G7  StatusBar：缺 changed 與 conflict 計數

設計（dc.html:1259-1261）：branch widget 之後依序是 ahead/behind、amber 方點＋changed 檔數、紅色 !N conflict 計數。

實作：有 branch＋ahead/behind＋incoming 指示；conflict 用「branch 名變紅」表達，無獨立計數；無 changed 計數。資料都已在 gitStore（changedPathSet、conflicted 清單），純呈現層補齊。

4. 已對齊、以及超出設計的部分

公平地說，M2 的底層與部分表面品質很好，重工時不需要動功能核：

  • ALIGNED Branch popup：288px、frost glass、14px 圓角、Local/Remote 分組、current check、ahead/behind、Fetch、New branch inline input——與 dc.html:1506-1542 幾乎逐項對齊。
  • BEYOND Checkout 保護（dirty tab 阻擋＋notice）、remotePaused 認證告警、ConflictBanner（abort/continue 的 in-progress 操作橫幅）、GitGuidedSetup（git 缺失時的引導）——設計稿沒畫，但都是正確的產品決策，保留。
  • BEYOND CodeMirror 6 diff：設計原型的 diff 是靜態假資料；實作用 MergeView／unifiedMergeView 提供真正可捲動、語法感知的 diff，能力超出設計。50MB／binary 防護也已就位。
  • ALIGNED Design tokens：styles.css 的 --yz-*／--paper-*／--ink-*／--line-* 體系與設計稿同源，Phase A–E 重工時 token 可直接沿用，不需要新調色。
  • ALIGNED 功能核：stage／unstage／discard／commit／fetch／pull／push／checkout／new branch 全部經 gitStore.runOp 落地並有測試。缺的是「擺在哪裡、長什麼樣」。

5. 為什麼會偏移（root cause）

M2 的 20 個 task 是以「Git 功能」為單位切的（porcelain、watch、操作 commands、askpass、conflict UX…），UI 只掛在讓功能可見的最小外殼上：tabs 預留了設計的三格骨架，但 Log／Console 被歸類成「之後的功能」而 disabled。問題在於——在設計稿裡 Log 不是一個附加功能，而是 Git mode 的預設門面；sidebar 的 commit 卡片也不是裝飾，而是設計的核心動線（quick commit 不離開 sidebar）。功能導向的切法把「呈現架構」降級成了流程尾端的殘餘工作，於是長出了一個功能正確、但資訊架構與設計稿不同的 Git mode。

對 M3 的教訓：各 milestone 開工前，先從設計稿抽出該 mode 的「表面清單＋預設視圖＋動線」作為驗收基準（就像本報告 §1 的表格），實作 task 對著表面切，而不是對著後端能力切。

6. 解決方案：五個 Phase 補齊呈現層

原則：不動功能核，只補呈現層與缺的資料管線。順序按「投資報酬＋依賴」排：A 便宜且立刻改變觀感 → B 便宜補一個 tab → C 是核心工程（唯一需要動 Rust）→ D 依賴 C 的 commit 資料 → E 牽涉 G6 的收斂決策放最後。A＋B 可以同一個 wave 並行；C 的 Rust 端與前端 graph 演算法也可並行。

Phase A — Git surface 骨架對齊 規模：小

  • A1：GitPanelTabs tab strip 右側加 branch pill（複用 BranchPopover，anchor 換到 pill）＋ Fetch／Pull／Push 三顆 icon 鈕（直接呼叫 runOp，busy 狀態沿用現有 op 鎖）。
  • A2：Local changes 的 diff header 補語言標籤與 +N／−N 統計（資料可由 diff hunks 計算，或 porcelain numstat）。
  • A3：StatusBar 補 amber changed 計數與紅色 !N conflict 計數（資料已在 gitStore）。
  • A4：Local changes tab 標籤加 amber 計數 pill（dc.html:730 樣式）。
驗收：與 dc.html:724-741、1259-1261 逐項對照；Fetch/Pull/Push 從 Git surface 一擊可達。

Phase B — Console tab 規模：小～中

  • B1：gitStore 加 operation log ring buffer（欄位：cmd 顯示字串、輸出行、tone ok/err、時間戳；上限例如 200 條）；runOp 統一寫入，錯誤也記錄。
  • B2：Console tab UI：深色面（--term-bg）、$ cmd 綠字＋右對齊時間、縮排輸出、空狀態文案。唯讀。
  • B3：啟用 Console TabsTrigger。
驗收：執行 pull/push/commit 後切到 Console 能看到指令與輸出，樣式對照 dc.html:918-932。

Phase C — Log tab（核心） 規模：大

  • C1（Rust）：git_service.rs 新增 log 管線：分頁 commit 清單（hash、subject、author、email、date、parents、refs）＋單 commit 的 stats／changed files。建議 cursor-based 分頁（例如每頁 200）＋ author 清單查詢供篩選器。
  • C2（前端）：graph lane 配置演算法（由 parents 拓撲算 lane/merge 曲線）＋ SVG 渲染，行高 32px 與列表對位；lane 色循環對照設計（藍/綠/橘）。虛擬捲動（清單可能上萬筆）。
  • C3：filter toolbar：文字（message/author/hash）＋ User 下拉＋ Date 篩選；篩選時 graph 可簡化為直線模式（JetBrains 同款妥協）。
  • C4：commit details 右欄 240px：hash chip＋copy、subject、avatar（initials）、email、date、parents、changed files（點擊 → Phase D modal）。
  • C5：操作鈕：Checkout（複用現有 checkout ＋ dirty 保護）；Compare（依賴 D）；Cherry-pick／Reset 建議入 M3 backlog（設計稿中 Reset 本身就是 disabled 樣式，dc.html:861）。
  • C6：預設 tab 改回 log（對齊 dc.html:1625），並啟用 Log TabsTrigger。
驗收：切入 Git mode 第一眼是 commit graph＋列表＋details；篩選、選取、checkout 可用；對照 dc.html:743-868。

Phase D — Diff viewer modal 規模：中

  • D1：把 DiffView 包進 1040px×88vh modal（shadcn Dialog）：header（serif 標題＋來源 sub＋Unified/Split toggle＋close）、左 236px 檔案清單、右 diff。
  • D2：接三個入口——sidebar「Review diff」（working tree 全部檔案）、Log details 檔案點擊／Compare（commit diff，需 C1 資料）、editor context menu「與 HEAD 比較」（openDiffCurrent 動線）。
驗收：三個入口都開得出 modal；unified/split 可切；對照 dc.html:1393-1465。

Phase E — Sidebar Git nav 重建 規模：中

  • E1：頂部 commit 卡片：branch pill＋ahead/behind＋changed pill＋commit input＋「Commit N／Review diff／hold-to-discard」（hold 交互已有設計規格：pointer down 起算、填滿動畫）。
  • E2：STAGED／CHANGED 逐檔清單（抽出 LocalChangesTab 的 FileRow／GitBadge／SectionHeader 共用）。
  • E3：依 §7 的 D1 決策收斂主區 local tab 左欄（移除 commit box、改平面清單，或保留現狀）。
驗收：不離開 sidebar 完成 stage→輸入訊息→commit；對照 dc.html:177-243。

建議波次：Wave 1＝A∥B → Wave 2＝C（Rust 與前端 graph 可再拆並行）→ Wave 3＝D∥E（E3 等 D1 決策）。全程不需改動 gitStore 既有 API 形狀，只增不改。

7. 需要你裁決的四件事

D1  Commit box 的最終位置

設計＝sidebar 卡片內；實作＝主區 Local changes 左欄底部。選項：(a) 完全依設計搬回 sidebar，主區左欄改為設計的平面清單；(b) 兩處並存；(c) 維持現狀。

建議：(a)。你已明確要求「依照 design 的方式」，且設計的分工（sidebar 管 staging＋commit、主區管瀏覽）動線更乾淨；(b) 會有兩個訊息輸入框的狀態同步問題。

D2  Local changes 右欄的 Unified/Split toggle 去留

設計中 local tab 是 unified-only（split 只在 Diff modal）；實作已有 toggle 且好用。

建議：保留 toggle（超出設計但無衝突），僅補上缺的 lang＋+N/−N。若要像素級對齊設計，則 toggle 移入 Phase D 的 modal。

D3  Log 危險操作的範圍

設計 details 面板有 Checkout／Compare／Cherry-pick／Reset 四鈕，其中 Reset 在原型中即為 disabled 樣式。

建議：M2.5/M3 先落 Checkout＋Compare；Cherry-pick 與 Reset（hold-to-confirm）入 M3 backlog，跟 conflict UX 一起驗。

D4  補齊工作的里程碑歸屬

五個 Phase 約當一個「M2.5 呈現層對齊」小里程碑，或併入 M3 開頭。

建議：獨立 M2.5 執行 A＋B（小、立即見效），C＋D＋E 視 M3 排程決定是否前置——C 是使用者感知最強的一塊，愈早愈好。
