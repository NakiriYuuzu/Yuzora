TL;DR

yuzora 是一個 local-first IDE workbench（Tauri 2 ＋ React 19 ＋ CodeMirror 6 ＋ Rust services），在 2026-07-02 → 07-10 的九天內，以「spec 先行 → 多代理波次實作 → 對抗式 review → 實機 GUI 驗收」的流程從零推進到 MVP＋。五個 mode（Files／Git／Database／SSH／AgentZone）加 editor、terminal、preview、LSP、logs 全部落地，前端 31.8k 行、Rust 22.6k 行、測試近 1:1（29.4k 行）。

架構體質良好：邊界原則清楚（IDE 管 workspace，agent runtime 自管 MCP／memory）、105 個 IPC command 的薄契約層、22 個 zustand store ＋ 9 個 Bridge 的事件驅動前端、每個關鍵選型都有書面論證與 spike 驗證。風險集中在四處：手寫 IPC 型別契約無 codegen（drift 風險）、安全縫（csp: null、SFTP 無 client 端 containment、SSH TOFU 無首次確認）、Windows 平台債（askpass stub、Job Objects 未實作、CI 不編譯 Windows）、以及 160 項未提交異動混雜三個主題。建議依 07-10 分析報告的 RC1 收斂路線：先拆分提交、補契約與平台防線，再開新功能。獨立覆核（見末頁分頁）另補兩個高槓桿缺口：發布就緒最小集（LICENSE／auto-updater／使用者文件）與持久化 versioning（現況 discard-only、無 migration）。

產品定位與邊界原則

創始 spec（docs/superpowers/specs/2026-07-02-yuzora-mvp-workbench-design.md）把 yuzora 定義為「能日常使用的核心工作流」優先的本機 IDE workbench，而非一次做完整 IDE 平台。最重要的架構決定是邊界宣言：

Boundary Principle（spec 原文改寫）
IDE 管 workspace、檔案、編輯、Git、terminal、Preview、LSP 與本機診斷；Agent runtime 自己管理 MCP、memory、tooling 與大型檔案 context。yuzora 明確不內建 codebase-memory-mcp、不管理 MCP server lifecycle、不替 agent 裝工具。這條邊界在 M5 落實為協定行為：session/new 的 mcpServers 一律傳空陣列。

五個 workspace mode 定義在 src/app/modes.ts：files / git / database / ssh / agent；Preview 是 editor 區的 singleton tab、Terminal 是全域 drawer、Settings 是 modal — 七個功能入口、三種層次，出自 UI Phase 的九條 architecture decisions（.superpowers/sdd/progress.md:15-25）。

持久化慣例是全庫一致的自訂模式：沒有任何一處使用 zustand persist middleware — 一律手寫 localStorage 讀寫＋try/catch＋白名單校驗。但覆核發現版本化並不一致：11 個持久化 key 中僅 2 個有內部 version 欄位（sessionIndex、workbenchLayout）、7 個靠 key 名 .v1 後綴、3 個完全無版本（agent／terminal／preview settings）；且既有版本檢查全是 discard-on-mismatch 而非 migration — schema bump 即靜默重置使用者資料。另一代價是八套幾乎重複的樣板，settingsStorage.ts:45-62 的 readJsonSetting/writeJsonSetting helper 只有部分模組採用。

Store 之間幾乎零橫向依賴，唯一例外是 agentStore import 了 terminalStore／uiStore／recentWorkspaces／sessionIndexStorage 四個模組（src/state/agentStore.ts:19,26-28）— agent 功能天生跨界，是日後拆分的最大阻力點。

Bridge pattern：事件接線的統一形狀

9 個 src/workbench/*Bridge.tsx（22～115 行）全是「render null 的 React 元件」：useEffect 內 listen() Tauri 事件 → store.getState().action() 命令式更新。用 React 元件而非裸模組的理由是白拿生命週期（mount/unmount 自動 listen/unlisten）與依賴陣列驅動的監聽重建（如 GitBridge 的輪詢間隔隨設定變動，GitBridge.tsx:42-49）。

優點
  • 每個事件域一個檔案，App.tsx 一覽無遺
  • 事件處理不進 React render cycle，不觸發無謂 re-render
  • 可單獨 render 測試（每個 Bridge 都有對應 test）
代價
  • getState() 命令式存取散佈 36 個非測試檔，無單一審計點
  • Bridge 數量隨事件域線性成長，缺統一 registry
  • externalChange 例外地拆成 Bridge＋Resolver 兩件，命名不對稱

IPC 契約層與宣告式命令模型

src/lib/ipc.ts（559 行）是「極薄」包裝：每個 command 一個具名函式、手寫參數與回傳型別、直接 return invoke(...)，不吞錯、不重試、不記 log；串流型 command（pty／search／lsp／dev-server）用 Tauri Channel 包 callback。src/lib/types.ts（476 行）與 Rust serde 型別是純手寫對應 — 無 ts-rs／specta codegen，Rust 端改欄位不會有編譯期警訊，是全案最明確的 drift 風險（詳見「評估」分頁）。

右鍵選單採宣告式模型：contextMenuModel.ts 定義 13 種 kind 的 discriminated union ＋ {id, label, availability, danger, executor} 命令物件；contextMenuDefs.ts（758 行）是註冊表 — 但它 import 了 9 個 store＋3 個 command 模組＋ipc，是全 repo import 面最廣的檔案，「新增一條選單命令 ≈ 要理解全 app store 拓撲」。

Editor / LSP / Terminal 客戶端的關鍵設計

  • documentRegistry／viewRegistry 是 module singleton 而非 store（src/editor/documentRegistry.ts:24-66）：entry 以 EditorView 物件身分守衛＋generation 計數器防 remount 競態（M1 驗收 FAIL-1 的 write-through＋generation guard 修法），這種細粒度身分語意不適合塞進 reactive store。已知取捨：generations map 刻意永不清除（跨 workspace 的微小常駐記憶體）。
  • Minimap 自製（src/editor/minimap.ts:10-53）：ViewPlugin 取樣行密度、MAX_BARS=2000 上限、rAF 合併重繪 —「建構絕不阻塞大檔打字」。
  • LSP 分段掛載（src/editor/EditorPane.tsx:175-195）：先同步掛無-LSP view（立即可打字），LSP extensions 之後經 StateEffect.appendConfig 併入。
  • Push/pull diagnostics 雙軌（src/lsp/diagnosticsPull.ts:1-11,71-74）：官方 serverDiagnostics() 服務 push 型 server；rust-analyzer 只答 pull，自寫 pull loop 且刻意不與 codeActions 的 lintSource 同掛（避免診斷雙渲染）。
  • Hover/completion HTML 一律過 DOMPurify（src/lsp/lspManager.ts:51-61）— LSP server 回傳任意 Markdown/HTML，是被明確處理的 XSS 面。
  • Terminal 延遲清理（src/terminal/TerminalSession.tsx:86-116）：setTimeout(0) 推遲 PTY 關閉、追蹤 in-flight openPromise — 專治 React StrictMode double-invoke 與 unmount 競態。
  • Preview 原生 webview 串行化（src/preview/previewCommands.ts:24-30,73-99）：child webview 是 process-wide singleton，所有操作走單一 promise chain；back/forward 只在能「證明導覽連續性」時才動真 webview。iframe 分支只允許 localhost（defence in depth，src/preview/PreviewFrame.tsx:10-20）。

並行模型與行程治理

  • 兩級狀態管理（lib.rs:42-93）：輕量狀態用 Mutex<Option/HashMap> 直包（絕不跨 await 持鎖）；長生命週期子系統用 Arc<Manager> 自管內部鎖（Lsp/Pty/Process/Agent×2/Ssh）。
  • DbHandle 三引擎三策略（db_service.rs:6-9,38-47）：SQLite＝std Mutex＋spawn_blocking；Postgres＝免鎖（client 本身 Send+Sync）；MSSQL＝tokio::Mutex（driver 要 &mut self）— 按驅動特性配鎖而非一刀切。
  • async 紀律：一律 tauri::async_runtime；只有 ssh/db 直接 named tokio（select!/timeout 不被 re-export，Cargo.toml 註解言明）。
  • 退出即殲滅（lib.rs:196-210）：RunEvent::Exit 時對 pty/process/agent_process/agent_terminal/ssh 全部 kill_all()＋preview stop_all()；kill_all 同時作為 Drop 路徑，防孤兒行程。

方案 A′：前端 host ACP client

M5 設計（docs/superpowers/specs/2026-07-06-yuzora-m5-agentzone-design.md §2.3）調查了 Zed／Superconductor／Orca 後裁定：ACP client 回呼本質是 UI 事務，在 Tauri 雙層架構中應住 webview。Rust host（方案 B）會自造 Zed 刻意避免的跨邊界 correlation 簿記，且官方 Rust crate 的 !Send futures 與 tokio runtime 整合彆扭。Rust 因此退為「薄管道」：agent_spawn/agent_write/agent_kill/agent_list ＋ stdout 逐行事件。

讀圖重點：JSON-RPC 的解析、session 狀態、permission UX 全部留在單一語言（TS）裡；跨 IPC 邊界流動的只有「原始行」與「行程生命週期事件」。

統一架構（07-07~08 rev.2）的三個核心修正

  • SDK 遷移：@zed-industries/agent-client-protocol → @agentclientprotocol/sdk@1.2.x（精確 pin），並用 acpConnection.deps.test.ts 以 ?raw 原始碼斷言防止 import 退化 — 用測試而非 lint 規則做防護欄。
  • agentRouter 以 (command, cwd) 為 key（src/agent/agentRouter.ts:18,34,58-65）：pi／claude／codex／custom 各自惰性建子連線，agentStore 只看到統一的 AgentConnection 介面；命令解析 TOCTOU-safe（route 算一次重用），custom command 以 SHA-256 指紋比對、原文不落盤。
  • Session 持久化走 ADR 0001：只存 Session Index（id／title／agentId／cwd／時間戳，每 workspace 20 筆），transcript 一律不落地，重啟靠 session/load replay 重建 — 拒絕了完整落地與完全不持久化兩個極端。

reduceSessionUpdate（acpConnection.ts:288-328）是純 switch reducer：message/thought/tool_call（以 toolCallId 合併 update）/plan 進 transcript；available_commands_update、session_info_update、usage_update 三個 side-channel 在 reducer 之前被攔截送 store callback；未知 variant 走「每型一次警告」的前向相容 default（SDK 的 zod 已擋真正畸形訊息）。

AgentZone 殘餘風險

  • MED ClientSideConnection 在 SDK 1.2.1 已標 @deprecated（建議改用 client() typed-handler API）— 目前功能正常，但 SDK 2.0 移除時會被迫遷移，宜排程。
  • MED Agent → Editor 硬耦合：writeTextFile/readTextFile 回呼直接 import getView／documentRegistry／saveFile（acpConnection.ts:11-14,396-427），沒有抽象層 — 這正是 M5 設計「fs 回呼走 buffer 層」的落實，行為正確，但依賴方向值得用介面固定下來。
  • LOW pi-acp 仍是 0.0.x 早期版本、auth 依賴非標準 authMethod 擴充欄位；已用假 agent fixture（src/agent/fakeAcpAgent.ts＋fixtures/fake-acp-agent-0b.mjs）隔離測試。

測試工程：規劃 vs 實際

Spec 規劃了 L1–L6 六層金字塔（unit → mock-IPC Playwright → tauri-driver 原生 E2E → macOS 手動 checklist → release smoke → 探索 QA）。實際落地的是 L1 加強版＋L4：

層規劃實際
L1 單元cargo ＋ vitest，每 PR超額 113 個前端測試檔（29k 行，≈1:1 code ratio）＋328 個 Rust #[test]；mockIPC 逐案例 stub；zustand 跨測試污染在 setup.ts 統一重置
L2 Playwright mock-IPCchromium＋webkit 每 PR未建
L3 tauri-driver E2ELinux＋Windows blocking未建
L4 macOS 手動腳本化 checklist制度化 每 milestone 一份 checklist ＋ gui-acceptance skill（computer-use 實機驅動、留證據截圖）— 把 jsdom 測不到的 watcher/tab/dialog/視覺全掛在這層

兩個值得推廣的自家發明：deps 防退化測試（用 ?raw import 斷言依賴 scope，取代 lint 防護欄）與 fixture 產生器家族（fixtures/gen-*.ts：大檔／encoding（與 Rust 測試逐位元組一致）／git 三態＋衝突 repo／LSP 四語言錯誤專案／preview 假專案），專供手動驗收復現。

工具鏈與 CI

  • bun 全家桶：install／test（vitest 4）／build（vite 8）；typecheck 用 @typescript/native（tsgo，typescript@7.0.2 preview）跑兩份 tsconfig — 快，但屬實驗性工具鏈，是「已知取捨」。
  • eslint 10 flat config：react-hooks v7 的 compiler 系規則降 warn（避免誤判 ref-as-cache 等既有模式）；no-undef 關閉（typescript-eslint 覆蓋）；無 no-restricted-imports 類防護欄（由 deps.test 代打）。
  • CI（.github/workflows/ci.yml）：frontend job（ubuntu：lint→typecheck→test→build）∥ rust job（只跑 macOS：fmt→clippy -D warnings→test）。缺口：Windows/Linux 的 #[cfg] 分支平時不被編譯，首次驗證落在 release tag。
  • Release（release.yml）：tag 版本與 tauri.conf.json 一致性守門 → 三平台矩陣（macOS universal／ubuntu／windows）→ draft release；macOS 簽章/notarization env 整段註解待 secrets。deploy-pages.yml 部署 site/ 產品頁。
