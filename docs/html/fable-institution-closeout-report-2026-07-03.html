制度地圖 — 常駐層極薄、細節按需載入、角色可直接派工

① 漏 token — always-on 注入肥大（每 session 起跑先燒 ~40–50KB）部分已修

實測：rules/ 6 檔 8,535B ＋ 舊 CLAUDE.md 4,302B 每 session 全載，git 規則重複四份（Co-Authored-By 出現 4 次）；superpowers SessionStart hook 每次注入 ~11KB；~120 個 skill 描述估 15–20KB；MCP 層 ~90 個 deferred tool 名 ＋ 指示。位元組成本還是小事，注意力稀釋才是大事——弱模型會給所有常駐指令權重，無關規則會稀釋相關規則的權威。

已修：rules/ 解散、git 四檔合併去重、co-founder 改按需載入、「新增常駐內容預設禁止」入 maintenance.md。待你裁決：閒置 skill 清理、superpowers plugin/hook 去留。

② 失焦 — 多套流程框架互搶、無優先序宣告已修

當日實例：yuzora 專案 CLAUDE.md（v1.0：禁 subagent、禁一切 git）與 SKILL.md v1.1（派 opus subagent、git 唯讀）同時生效互相矛盾，你只好手動調解。根因：同一條規則全文維護在兩個檔案，改了一邊。另外 superpowers「回應前必須 invoke skill」與 plan-first、yuzora-loop 閘門三方搶同一個任務。
