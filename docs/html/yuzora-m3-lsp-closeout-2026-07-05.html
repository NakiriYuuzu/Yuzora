重大 findings 精選

五則最能代表 M3 對抗 review 深度的 finding——問題 → 發現方式 → 根因修。每則的共同特徵：表面症狀之下有結構性根因，controller 拒絕半吊子修法、要求修根因不修引信。

R3-F1reload 失敗鏈的 keyed remount 抹掉未存 buffer（wave 5）major
問題wave 5 R2B-F1 修好了「reload-missing 讓 promise settle」後，一位全新 reviewer（w5r3-final）發現 settle 反而觸發 EditorArea 的 keyed remount（key=path:documentGeneration）：舊 pane cleanup 的 buffer 寫回被 generation guard no-op、新 pane getDocument reject 成空白 pane，dirty=true 卻無 view 可存——「re-save 重建」承諾落空。
發現fresh-context reviewer 對前一輪修復做專項攻擊，controller 親驗五宣稱（workspaceStore 無條件重建 groups → remount → gen-guard no-op → 空白 pane）全數成立。
根因controller 判定比 reviewer 深一層：documentRegistry 的 eager bump（先 delete＋bump generation 才 fetch）本身即 time bomb——bump 後 fetch 失敗，任何 re-render（git poll／LSP 事件）都會引爆同一鏈；reviewer 的近端方案只拆引信、time bomb 仍在。改為 bump-after-success（fetch 成功後才遞增 generation）。w5r4-dry 確認此為結構性正確而非時序僥倖：key 與 gen 同源，「該 remount 時 buffer 必 no-op／不該 remount 時 buffer 必存活」成為不變式。
W6A-F1一鍵安裝無視 workspace override → 裝錯 server 死路（wave 6）major · breaks-requirement
問題T14 的 lsp_install_server 用自寫 resolve_active，只讀 cfg.defaults → adapter default，從不讀 workspaces override。情境：scope 預設 workspace、override python=pylsp、卡片顯示 pylsp、點安裝卻裝了 pyright → pylsp 永遠 Missing＝死路，違反 A9'「切任一 profile 體驗一致」。
發現w6r1-rust reviewer 把 implementer 自報的偏差④判為不可接受；controller 親驗 resolve_active（:124-129）與 handler／ipc.ts 皆無 workspace 參數，而現成的 lsp_config::resolve_server（override→defaults，含測試）根本沒被 install 路徑呼叫。
根因「動作端」（install）與「生效端」（startup）解析邏輯分岔＝「裝的 ≠ 啟動會用的」。修法兩半、簽名契約由 controller 定死：前端 lspInstallServer(workspace, language) 傳 workspacePath；Rust handler ＋workspace: Option<String>，經與 set_server 寫入側同一 canonical_key() helper 正規化後走 resolve_server。install 的 resolve 鏈與 lsp_service 啟動 resolve 鏈至此完全同源（沉澱為 LEARNINGS 慣例）。
W6A-F2sync command 佔 main thread，安裝期餓死全部 IPC（wave 6）major
問題lsp_install_server 寫成非 async 的 #[tauri::command] pub fn，註解卻宣稱「off the UI thread」——與 Tauri v2 事實相反：non-async command 跑在 main thread，而 body 含 blocking ureq 下載＋npm／pip 子行程（分鐘級），安裝期間全部 IPC 餓死。
發現w6r1-rust reviewer 對照 Tauri v2 執行模型；違反 A2 明列裁決（「同步 blocking＋背景執行緒」）。
根因改 async fn＋tauri::async_runtime::spawn_blocking（以官方文件為準，InflightGuard 不跨 await、JoinError→Err 收斂）。同時揭露一條暗雷並沉澱為 LEARNINGS：本 codebase 其他 sync command（git_push／git_fetch 等）同屬此類，M3 範圍外未修、記 backlog。
M3F-2npm/pip 子程序無 timeout，hang 時永鎖安裝（close-out fable review）minor · latent
問題fable 最終 reviewer 發現 run_command（:456-465）以 .status() 無限等待子行程——npm／pip／venv 若 hang，InflightGuard 永鎖、前端進度永卡。fable 已核 W6A-F4（download 側 timeout）不涵蓋此處，判為真 gap 非重複。
發現fable fresh-context 審計時對照 git_service.rs（:87／:118 已有現成 deadline poll+kill 模式），指出 install 側漏用。
根因run_command 改 spawn＋deadline try_wait 輪詢＋逾時 kill+wait（逐字仿 git_service）；常數 NPM_PIP 600s／VENV 120s。測試 run_command_times_out_and_kills：sleep 30＋300ms timeout 於 0.34s 回 Err 證明真 kill。殘留：process-group kill（npm 孫程序孤兒）與 git_service 同批工程，記 backlog。
M3C-F1文件宣稱與 client cache 現實的兩輪校正（close-out）minor
問題fable 的 M3F-1 修復把 checklist 第 5 項改為「profile 切換後重開該語言檔案即生效」；但驗證輪（m3c-r2-verify）帶著關鍵疑點命中——這是假宣稱。
發現reviewer 追 lspManager client cache：鍵為 (workspace, language)（:50）、ensureClient cache-hit 直回舊 client（:66-67）、EditorPane unmount 不清 cache，僅 stopWorkspace（workspace 切換）／重啟清除。四路反證（unmount／transport 吞錯／setConfig 不動 workspacePath／redetect 不清 cache）皆不倒。
根因checklist 兩處措辭改為：新 profile 於 app 重啟、或切至別 workspace 再切回（觸發 stopWorkspace 清 cache）後生效；同 session 同 workspace 重開檔不生效（cache 以 (workspace, language) 為鍵）。最終驗證輪（m3c-r3-final）對五宣稱逐一對碼核驗 CLEAN。此為 review 對「文件正確性」同等對抗的示範——連修復引入的文件宣稱都要對碼複驗兩輪。

兩個 M3 期間定型的關鍵設計

雙診斷架構：push=serverDiagnostics ／ pull=gated linter

wave 4 對抗 review 糾正了一個事實錯誤：@codemirror/lsp-client 的 languageServerSupport 不含 serverDiagnostics——原「與 baseline serverDiagnostics 互補」的自報不成立，push-only server（如 typescript-language-server）現況會零診斷渲染。定型為：push 通道掛 serverDiagnostics()（LSPClient config，含 unsyncedChanges 補償）；pull 通道的 linter(lspLintSource) 僅於 diagnosticProvider 存在時掛（gate），並包 try/catch。both-capable server 的雙寫風險記 known limitation，實機驗證。

五態語意：stopped → syntaxOnly

LspDisplayState 由前端組合 process status＋client initialized＋檔案分級推導出五態：ready／starting／failed／missing／syntaxOnly。wave 5 B-F2 發現 deriveDisplayState 缺 stopped 分支——「profile 切換＋重新偵測」會讓 Stopped info＋initialized=true 落到 ready，讓死 server 顯示綠 Ready。修法在維持五態 vocabulary 前提下加明確分支 stopped → syntaxOnly（誠實呈現「無活 LSP」），不新增第六態。其餘 lifecycle 缺口（set_server stop 不 emit／initialized 無 per-language reset／profile 切換不重掛 client）記 backlog。

LEARNINGS 沉澱（M3 期間新增）

M3 對抗 review 過程中被獨立 reviewer 或 controller 撞出的陷阱與慣例，已寫入 .claude/skills/yuzora-loop/LEARNINGS.md（只增不刪、與 SKILL 衝突時取較嚴者）。以下為 2026-07-04／07-05 新增條目。

日期 · 類型沉澱來源
07-04 tighten不可信 HTML 注入有全域 CSS 框架（Tailwind）的 body portal 時，sanitizer 注入面窮舉必含應用層屬性 class/id（攻擊者可用已編譯的 utility class 重建 overlay）；且逐屬性黑名單是在跟 CSS 賽跑，應優先做 CSS 定位模型根因防禦（contain:paint 建 containing block，一次斷 position:fixed 逃逸）。wave 2 R9→R11
07-05 trapsubagent Write 可能夾入 NUL byte（0x00）——檔案被 grep/rg/file 判 binary 而對程式碼搜尋隱形，tsc/vitest 照綠、git 仍當 text diff；每輪修復後掃全 src。wave 4 R2-1
07-05 tighten修復批次的單檔 vitest 驗證不夠——controller 整合驗證必跑全量 vitest 並檢查「Errors／Unhandled Rejection」行；N passed 但 Errors 1 不算綠（跨測試檔交互如「新 production 依賴未在別檔 mock」只在全量跑才現形）。wave 4 批次 C
07-05 tightenasync IIFE 的 unhandled-rejection 修復必須窮舉該 IIFE 全部 reject 源（await 鏈逐個列：getDocument／request／initializing…），只補單一 .catch 會留下「保證不完整」。wave 4 R3-1
07-05 trapkeyed-remount 生態（key=path:generation）中「先 bump 再 fetch」的 eager bump 是資料遺失 time bomb——世代號只可在新內容真正到位後遞增，否則 fetch 失敗後任何 re-render 都會 remount 掉活 pane、cleanup 的 write-back 被 gen guard no-op 而永久丟失未存 buffer。wave 5 R3-F1
07-05 tighten裁決修法時修根因不修引信——reviewer 首選方案「不清 flag 避開 re-render」只拆近端引信（危險狀態仍在，任何他源 re-render 照樣引爆）；controller 必問「此修之後，原危險狀態本身還存在嗎」，答存在即改修根因。wave 5 R3-F1
07-05 convention測試要斷言某模組的內部行為（如 generation bump 時序）時不得 mock 該模組本身，改 mock 其 IO/IPC 邊界（真 reloadDocument＋mock ipc.openFile）——否則斷言測的是 mock 而非實作，綠燈無證據力。wave 5 R3-F1
07-05 trapTauri v2 非 async 的 #[tauri::command] 跑在 main thread——含阻塞 IO（下載／子行程／網路 git）的 command 會餓死全部 IPC；長阻塞 command 必 async fn＋spawn_blocking。wave 6 W6A-F2
07-05 tighten「動作端」與「生效端」的解析邏輯必須同源——install 用自寫 resolve（只讀 global default）而啟動用含 workspace override 的 resolve，兩鏈分岔即「裝的 ≠ 啟動會用的」死路；review rubric 對雙端功能必問「兩端是否呼叫同一 resolve 函數＋同一 canonical 化」。wave 6 W6A-F1
另有 wave 2–3 期間關於 ultracode workflow 的操作型沉澱（內層 parallel() 必傳 thunk 非 promise；dry-pass 判定須核所有 lens 完成；opus 額度退化期單 agent 比 workflow 易過），以及「派修 brief 應授權 implementer『指示若型別／執行期不安全須停下回報並改採安全等價』」——後者在 wave 3 C2 攔下一次 controller 拍腦袋指令釀 runtime crash（@codemirror/lint@6 的 LintSource 型別不含 null）。
